OpenAI 三套 API 完整讲解

Chat Completions / Responses / Agents SDK,逐行讲解 + 最佳实践 + 与 Anthropic 的横向对照。写给会 HTTP/SDK 的资深程序员。主线 Python 官方 SDK openai 与 openai-agents。示例模型用官方 README 当前示例 gpt-5.5(请按你的可用模型替换)。

OpenAI 现在有三个层级的接口,很多人混用导致踩坑:Chat Completions 是经典的、无状态的、跨厂商事实标准;Responses 是新一代、有状态、内置工具与推理、流式是语义化事件;Agents SDK 是架在 Responses 之上的多智能体编排框架(Agent + Runner + handoffs + guardrails + sessions + tracing)。本文把三套的请求/响应/流式/工具/结构化输出全部拆开,并不断和 Anthropic 的对应概念对照,方便你迁移心智。

关于版本与命名。OpenAI 的模型名(如 gpt-5.5)、内置工具的 type 字符串、以及 .parse 是否还在 beta 命名空间,都在快速演进。本文锁定形状与心智模型(这些稳定),具体的模型字符串和工具版本号请以你写代码当天的官方文档为准。凡是易变处都会标注。

2Chat Completions API

从最经典的讲起。它是无状态的「一进一出」:你把整段对话(角色数组)发上去,拿回一条助手消息。生态里大量第三方接口复刻了它的形状,所以即使你主用 Responses,也得懂它。

2.1 客户端与最小请求

chat_basic.pypython
from openai import OpenAI

client = OpenAI()                       # 从环境变量 OPENAI_API_KEY 读取密钥

completion = client.chat.completions.create(
    model="gpt-5.5",
    messages=[
        {"role": "developer", "content": "你是简洁的编码助手。"},
        {"role": "user", "content": "怎么判断一个 Python 对象是某个类的实例？"},
    ],
)

print(completion.choices[0].message.content)
print(completion.choices[0].finish_reason)   # "stop"
print(completion.usage.total_tokens)
  1. OpenAI() 不传 key 是推荐写法,SDK 从环境变量 OPENAI_API_KEY 读取。别硬编码密钥。还可配 OpenAI(timeout=..., max_retries=...)(SDK 默认重试 429/5xx)。
  2. messages 是角色数组。每条 {"role", "content"}。这与 Anthropic 的 messages 几乎一致,最大差别:OpenAI 的系统提示放在 messages 里(一条 system/developer 消息),而 Anthropic 是顶层 system 参数。
  3. choices[0].message.content 取回复文本。choices 是数组(历史上支持 n>1 生成多条),默认取第 0 条。
  4. finish_reason 是 Chat 版的「停止原因」(类比 Anthropic 的 stop_reason),取值见 2.3。

2.2 messages / roles:system 还是 developer

角色有四种:system / developer / user / assistant(工具结果用 tool,见 2.5)。一个容易踩的点:

developer vs system。较新的模型(尤其推理/o 系列与 GPT-5 家族)推荐用 developer 角色承载「开发者指令」,它在指令优先级体系里高于 user、低于平台。旧模型用 system。多数情况下两者可互换;写新代码用 developer,迁移老代码看模型文档。Anthropic 没有这个区分(只有顶层 system + 4.8 的会话内 system 消息)。

2.3 响应结构 / finish_reason / usage

finish_reason含义
stop正常结束(或命中 stop 序列)
length撞到 max_tokens/max_completion_tokens,输出被截断
tool_calls模型要调函数,去执行并回填(2.5)
content_filter被内容过滤拦截

completion.usage 带 prompt_tokens / completion_tokens / total_tokens。推理模型还会在 completion_tokens_details.reasoning_tokens 里报告思考用量——它计费但不返回内容。

推理模型上别再用 max_tokens,改用 max_completion_tokens(它同时涵盖推理 token + 可见输出);max_tokens 在这些模型上被弃用。

2.4 流式与增量 arguments

stream=True 后,响应变成一串 chunk,每个 chunk 的 choices[0].delta 带增量。文本增量在 delta.content;工具调用的参数也是增量字符串——这正是和 Anthropic input_json_delta 一样的设计。

chat_stream.pypython
stream = client.chat.completions.create(
    model="gpt-5.5",
    messages=[{"role": "user", "content": "写一首关于递归的俳句"}],
    stream=True,
    stream_options={"include_usage": True},     # 末尾追加一个带 usage 的 chunk
)

tool_args = {}                                  # index -> 累积中的 arguments 字符串
for chunk in stream:
    delta = chunk.choices[0].delta if chunk.choices else None
    if delta and delta.content:
        print(delta.content, end="", flush=True)            # 文本增量
    if delta and delta.tool_calls:                          # 工具调用增量
        for tc in delta.tool_calls:
            buf = tool_args.setdefault(tc.index, {"name": "", "args": ""})
            if tc.function.name:
                buf["name"] = tc.function.name              # 名字只在首个 chunk 出现
            if tc.function.arguments:
                buf["args"] += tc.function.arguments        # arguments 是碎片，累积
  1. delta.content 是文本碎片,直接拼接打印。
  2. 工具调用按 tc.index 累积。和 Anthropic 一样:函数名只在第一个 chunk 给一次,之后 function.arguments 是 JSON 字符串碎片,你按 index 拼起来,流结束后再 json.loads。这就是「为什么没有 tool_call 整体 delta」——变化的只有 arguments 这个字段。
  3. stream_options={"include_usage": True} 才会在最后给一个带 usage 的 chunk(默认流式不报 usage)。这一点常被忽略。
  4. chunk.choices 可能为空(比如最后那个 usage-only chunk),要先判空再索引。

2.5 函数调用循环

这是 Chat Completions 的核心循环,和我们在 Anthropic 教程里手写的 agentic loop 同构,只是字段名不同。

chat_tools.py — 完整函数调用循环python
import json

tools = [{
    "type": "function",
    "function": {                                 # Chat 版：schema 嵌在 "function" 下
        "name": "get_weather",
        "description": "查询某城市当前天气",
        "parameters": {
            "type": "object",
            "properties": {"city": {"type": "string"}},
            "required": ["city"],
            "additionalProperties": False,
        },
        "strict": True,                           # 保证参数严格符合 schema
    },
}]

messages = [{"role": "user", "content": "北京现在天气怎么样？"}]

while True:
    resp = client.chat.completions.create(
        model="gpt-5.5", messages=messages, tools=tools,
    )
    msg = resp.choices[0].message
    messages.append(msg)                          # 关键：把整条助手消息追加回历史

    if not msg.tool_calls:                         # 没有工具调用 = 收尾
        print(msg.content)
        break

    for call in msg.tool_calls:                    # 可能并行多个
        args = json.loads(call.function.arguments)  # arguments 是 JSON 字符串，必须解析
        result = lookup_weather(args["city"])       # 你的实现
        messages.append({
            "role": "tool",                         # 工具结果是一条 role=tool 的消息
            "tool_call_id": call.id,                # 必须对上 call.id
            "content": result,
        })
  1. 工具定义形状:{"type":"function","function":{name,description,parameters,strict}}——注意 schema 嵌在 "function" 键下(这是 Chat 与 Responses 的一个关键差异,Responses 是扁平的,见 3.6)。tool_choice 可选 "auto"/"none"/"required"/{"type":"function","function":{"name":...}}。
  2. messages.append(msg) 把整条助手消息(含 tool_calls)追加回去,否则下一轮模型看不到自己发的调用。等价于 Anthropic 里「先追加 resp.content」。
  3. call.function.arguments 是 JSON 字符串,不是对象——必须 json.loads。这是和 Anthropic block.input(SDK 已解析成 dict)的一个差异:Chat 给你的是字符串。
  4. 每个结果是一条 role="tool" 消息,带 tool_call_id 对上调用的 id。注意和 Anthropic 的差别:Anthropic 把所有 tool_result 放进一条 user 消息;OpenAI 是每个工具结果一条独立的 tool 消息。
  5. 循环以 finish_reason/tool_calls 为信号:没有 tool_calls 就收尾。失败时把错误文本放进 tool 消息回填,让模型自愈(可加自定义字段说明)。

2.6 结构化输出

两种力度:response_format 约束整段回复为某个 JSON Schema;或用 SDK 的 .parse() + Pydantic 直接拿到类型化对象。

chat_structured.pypython
from pydantic import BaseModel

class Contact(BaseModel):
    name: str
    email: str
    plan: str

# 方式 A：SDK 帮你下发 schema 并解析（推荐）
resp = client.chat.completions.parse(           # 较新 SDK；早期在 client.beta.chat.completions.parse
    model="gpt-5.5",
    messages=[{"role": "user", "content": "抽取：张三 zhangsan@x.com 企业版"}],
    response_format=Contact,                     # 直接传 Pydantic 类
)
contact = resp.choices[0].message.parsed         # 已校验的 Contact 实例

# 方式 B：手写 json_schema
resp = client.chat.completions.create(
    model="gpt-5.5",
    messages=[...],
    response_format={
        "type": "json_schema",
        "json_schema": {
            "name": "contact",
            "schema": {
                "type": "object",
                "properties": {"name": {"type": "string"}, "email": {"type": "string"}},
                "required": ["name", "email"],
                "additionalProperties": False,
            },
            "strict": True,
        },
    },
)

strict: True + additionalProperties: False + 所有字段进 required 是结构化输出的硬性要求(可选字段用 "type": ["string","null"] 表达)。被拒(安全)或截断(length)时输出可能不符合 schema,要判 finish_reason。

3Responses API 新一代

3.1 为什么有 Responses:与 Chat 的形状差异

Responses 不是 Chat 的小改款,而是为「Agent 时代」重设计的接口。核心差异:

Chat CompletionsResponses
系统提示messages 里一条 system/developer顶层 instructions 参数
输入messages 角色数组input:字符串或 item 列表
输出单条 messageoutput:异构 item 列表(message/reasoning/function_call/web_search_call…)
取文本choices[0].message.contentoutput_text(便捷聚合)
状态无,自带历史store=True 存服务端,previous_response_id 串接
内置工具无web_search/file_search/code_interpreter/computer/image/MCP
函数工具形状嵌在 "function" 下扁平:type/name/parameters 同级
函数结果回填role:"tool" 消息type:"function_call_output" item
流式token 级 delta语义化 response.* 事件

3.2 最小请求:input / instructions / output_text

resp_basic.pypython
from openai import OpenAI
client = OpenAI()

resp = client.responses.create(
    model="gpt-5.5",
    instructions="你是简洁的编码助手。",       # 顶层系统指令（取代 system 消息）
    input="怎么判断一个 Python 对象是某个类的实例？",   # 可以是一个纯字符串
)

print(resp.output_text)         # 便捷属性：把所有文本输出拼起来
print(resp.id)                  # resp_...，用于下一轮 previous_response_id
  1. instructions 是顶层系统指令,等价于 Anthropic 的顶层 system。它每轮都可重设;若用 previous_response_id 串接,新一轮的 instructions 不会自动继承,要按需重传。
  2. input 可以是字符串(单轮最省事),也可以是 item 列表(多轮/多模态/工具结果,见下)。
  3. resp.output_text 是把所有文本 item 聚合后的便捷字符串。要精细处理就遍历 resp.output(3.3)。
  4. resp.id(形如 resp_...)是服务端这次响应的句柄,下一轮传给 previous_response_id 就能续上,无需重发历史(3.4)。

多模态/多轮时,input 用 item 列表,文本与图片用带 type 的内容块:

input 作为 item 列表python
input=[
    {"role": "user", "content": [
        {"type": "input_text",  "text": "这张图里是什么？"},
        {"type": "input_image", "image_url": "https://example.com/a.png"},
    ]},
]

3.3 输出是「item 列表」,不是一条消息

这是 Responses 最需要转变心智的地方。resp.output 是一个异构列表,每个 item 有 type。常见类型:

item type含义 / 关键字段
message助手文本,content 里是 output_text 部分(带 text、可能带 annotations 引用)
reasoning推理模型的思考项(默认只给摘要/加密),多轮要原样保留(3.7)
function_call你的自定义工具调用:name、arguments(JSON 字符串)、call_id
web_search_call / file_search_call / code_interpreter_call内置工具的调用记录(服务端已执行)
image_generation_call / mcp_call 等对应内置工具的产物
遍历 outputpython
for item in resp.output:
    if item.type == "message":
        for part in item.content:
            if part.type == "output_text":
                print(part.text)
    elif item.type == "function_call":
        print("要调用：", item.name, item.arguments, item.call_id)
    elif item.type == "reasoning":
        pass            # 推理项：原样留着，多轮回传

类比 Anthropic:Responses 的 output item 列表 ≈ Anthropic resp.content 的内容块列表(text / tool_use / thinking)。区别是 OpenAI 把工具调用、推理、内置工具结果都做成顶层 item,而 Anthropic 是内容块。

3.4 有状态:previous_response_id 串接 关键差异

Chat 是无状态的,你每轮自带全量历史。Responses 默认 store=True,把这次响应存在服务端;下一轮只发新输入,用 previous_response_id 指回上一条,服务端自动接上完整历史。

resp_multiturn.py — 不重发历史python
r1 = client.responses.create(model="gpt-5.5", input="金门大桥在哪个城市？")
print(r1.output_text)

r2 = client.responses.create(
    model="gpt-5.5",
    previous_response_id=r1.id,               # 接上 r1 的完整上下文
    input="那它在哪个州？",                     # 只发新问题，不重发前面
)
print(r2.output_text)
  1. previous_response_id=r1.id 让服务端把 r1 及其之前的历史作为上下文,你只发增量。省带宽、省拼接,但历史存在 OpenAI 服务端。
  2. store=False 可关闭存储(零留存/合规场景),但那样就不能用 previous_response_id 串接,退回 Chat 式自带历史。
  3. Conversations API 是更上层的封装:创建一个 conversation 对象,把多次 responses 挂在同一会话下,适合长对话/多端同步。
  4. 对照 Anthropic:Anthropic 没有服务端会话句柄,多轮一律自带 messages(或上 Managed Agents 的 session)。Responses 的 previous_response_id 是 OpenAI 这套独有的便利。

3.5 内置工具(服务端执行,声明即用)

Responses 自带一批 Anthropic「服务端工具」式的能力,声明在 tools 里,OpenAI 替你执行,结果作为 item 回到 output。

resp_builtin.pypython
resp = client.responses.create(
    model="gpt-5.5",
    input="纽约今天天气怎么样？查一下。",
    tools=[
        {"type": "web_search"},                                    # 联网搜索
        {"type": "file_search", "vector_store_ids": ["vs_123"]},   # 检索向量库
        {"type": "code_interpreter", "container": {"type": "auto"}},# 沙箱跑代码
        # {"type": "image_generation"} / {"type": "computer_use_preview"} / MCP ...
    ],
)
print(resp.output_text)
  • web_search:联网,结果带引用注解(annotations)。≈ Anthropic web_search。
  • file_search:在你的 Vector Store 里做 RAG 检索,要先建向量库并传 vector_store_ids。
  • code_interpreter:沙箱里跑 Python(数据分析、画图、生成文件)。≈ Anthropic code_execution。
  • MCP:用 {"type":"mcp", "server_label":..., "server_url":..., ...} 直接接远程 MCP 服务器。

内置工具的 type 字符串(如 computer_use_preview、code_interpreter 的 container 形状)会随版本变化,以当天文档为准。它们和你的自定义函数工具可以混在同一个 tools 数组里。

3.6 函数调用与 call_id 回填

自定义函数工具在 Responses 里是扁平形状,回填用 function_call_output item(不是 Chat 的 role:"tool" 消息)。

resp_function.py — 函数调用循环python
import json

tools = [{
    "type": "function",            # 扁平：name/parameters 与 type 同级（无 "function" 嵌套）
    "name": "get_weather",
    "description": "查询某城市当前天气",
    "parameters": {
        "type": "object",
        "properties": {"city": {"type": "string"}},
        "required": ["city"],
        "additionalProperties": False,
    },
    "strict": True,
}]

history = [{"role": "user", "content": "北京天气？"}]
resp = client.responses.create(model="gpt-5.5", input=history, tools=tools)

# 找出所有 function_call item，执行并回填
outputs = []
for item in resp.output:
    if item.type == "function_call":
        args = json.loads(item.arguments)             # arguments 是 JSON 字符串
        result = lookup_weather(args["city"])
        outputs.append({
            "type": "function_call_output",           # 回填项类型
            "call_id": item.call_id,                  # 对上 function_call 的 call_id
            "output": result,                         # 字符串
        })

# 把模型这轮的输出 + 工具结果接着发回（或用 previous_response_id 省去 resp.output）
resp2 = client.responses.create(
    model="gpt-5.5",
    previous_response_id=resp.id,                     # 接上上文
    input=outputs,                                    # 只发工具结果
    tools=tools,
)
print(resp2.output_text)
  1. 扁平工具形状:{"type":"function","name":...,"parameters":...,"strict":True}——和 Chat 的「嵌在 function 下」不同。把 Chat 代码搬过来时这里最容易报错。
  2. 回填用 function_call_output item,带 call_id(对上 function_call.call_id)和 output(字符串)。这取代了 Chat 的 role:"tool" 消息。
  3. 有两种续法:① 用 previous_response_id 接上文,input 只放 function_call_output(推荐,省事);② 不用状态,则把 resp.output 整段 + 工具结果一起塞进新 input(类似 Chat 自带历史)。
  4. 对照 Anthropic:call_id = tool_use_id;function_call_output = tool_result。心智完全一致,只是命名和「item vs 内容块」的容器不同。

3.7 推理模型与 reasoning.effort

GPT-5 / o 系列是推理模型。Responses 用 reasoning 参数控制思考。

resp_reasoning.pypython
resp = client.responses.create(
    model="gpt-5.5",
    input="证明：任意 5 个连续整数之和能被 5 整除。",
    reasoning={
        "effort": "medium",        # minimal / low / medium / high（视模型而定）
        "summary": "auto",         # 让服务端返回一段推理摘要
    },
)
# 推理摘要在 output 里的 reasoning item / 或 resp 上的便捷字段（按 SDK 版本）
print(resp.output_text)
  1. reasoning.effort 控制思考深度与花销(≈ Anthropic 的 output_config.effort)。值通常是 low/medium/high,部分模型还有 minimal。深的更准但更慢更贵。
  2. reasoning.summary="auto" 才会给可读的推理摘要;原始思维链不返回(和 Anthropic 一致)。不设就只看到结果。
  3. 推理项要保留。多轮(尤其带工具)时,把 output 里的 reasoning item 原样回传能显著提升连贯性。用 previous_response_id 串接时服务端自动保留;手动自带历史时别把 reasoning item 丢了。
  4. 推理 token 计费但不可见,在 usage 里有计数。别用 temperature 等采样参数去「控制」推理模型——很多推理模型不接受这些参数。

3.8 结构化输出

和 Chat 同理,但参数在 text.format 下(不是 response_format),或用 responses.parse + text_format。

resp_structured.pypython
from pydantic import BaseModel

class Step(BaseModel):
    explanation: str
    output: str
class Solution(BaseModel):
    steps: list[Step]
    answer: str

# 方式 A：parse + Pydantic（推荐）
resp = client.responses.parse(
    model="gpt-5.5",
    input="一步步解 8x + 7 = -23",
    text_format=Solution,
)
sol = resp.output_parsed            # 已校验的 Solution 实例

# 方式 B：手写 json_schema
resp = client.responses.create(
    model="gpt-5.5",
    input="...",
    text={"format": {
        "type": "json_schema",
        "name": "solution",
        "schema": { "type": "object", "properties": {...},
                    "required": [...], "additionalProperties": False },
        "strict": True,
    }},
)

记差异:Chat 用 response_format= + message.parsed;Responses 用 text.format / text_format= + output_parsed。json_object(只保证是 JSON、不校验 schema)在两者都有,但能用 json_schema + strict 就别用它。

3.9 语义化流式事件 CORE

Responses 的流式不是 Chat 那种「一坨 delta」,而是语义化的命名事件——和 Anthropic 的 content_block_* 思路如出一辙。设 stream=True,迭代得到 response.* 事件。

resp_stream.pypython
stream = client.responses.create(
    model="gpt-5.5",
    input="写一段关于流式的科普",
    stream=True,
)
for event in stream:
    if event.type == "response.output_text.delta":
        print(event.delta, end="", flush=True)            # 文本增量
    elif event.type == "response.function_call_arguments.delta":
        accumulate(event.delta)                           # 工具参数增量（字符串碎片）
    elif event.type == "response.reasoning_summary_text.delta":
        pass                                              # 推理摘要增量
    elif event.type == "response.completed":
        final = event.response                            # 完整 Response 对象在这里

常见事件类型(按生命周期):

事件 type含义
response.created / response.in_progress响应开始 / 进行中(壳对象)
response.output_item.added新增一个 output item(message / function_call / reasoning…)
response.content_part.added某个 message item 内新增一个内容部分
response.output_text.delta / .done文本增量 / 该文本部分完成
response.function_call_arguments.delta / .done函数参数 JSON 字符串增量 / 完成(类比 input_json_delta)
response.reasoning_summary_text.delta推理摘要增量
response.output_item.done某个 item 完成
response.completed / .failed / .incomplete整体结束 —— event.response 是完整 Response
error流中错误
  • 对照 Anthropic:response.output_text.delta ≈ text_delta;response.function_call_arguments.delta ≈ input_json_delta(同样是参数 JSON 字符串碎片,累积后解析);response.output_item.added ≈ content_block_start;response.completed ≈ 末尾 message_delta+message_stop,且直接带完整对象。
  • response.completed 里有完整 response,等于 Anthropic 的 get_final_message():就算你只想要终态,流式也是高 max_tokens 下避免 HTTP 超时的正解。
  • 未知事件优雅忽略:事件类型会持续新增,别对未识别的 type 崩溃。

4Agents SDK 框架

前面两套是「打一次 API」。Agents SDK(pip install openai-agents)是编排框架:它替你写好了 agentic loop、工具循环、多智能体路由、护栏、会话记忆和追踪。它默认跑在 Responses 之上。心智上,它就是我们在 Anthropic 教程里手写的那个循环——被产品化了。

4.1 心智模型:Agent + Runner 的 agent loop

两个核心对象:Agent(配置:指令、模型、工具、护栏、handoff)和 Runner(执行器:跑那个循环)。Runner 的循环:

Runner 的 agent loop（官方语义）flow
调用 LLM（当前 agent + 当前输入）
  ├─ 产生 final_output 且无工具调用  → 循环结束，返回 RunResult
  ├─ 发生 handoff                   → 切换 current agent，重启循环
  └─ 有工具调用                     → 执行工具，把结果追加进输入，再循环
超过 max_turns                      → 抛 MaxTurnsExceeded（max_turns=None 关闭）

这正是我们手写过的循环,只是 Runner 把「判停止原因 / 追加历史 / 执行工具 / 回填」全包了。

4.2 定义 Agent 与 Hello World

hello_agent.pypython
import asyncio
from agents import Agent, Runner

agent = Agent(
    name="History Tutor",
    instructions="清晰简洁地回答历史问题。",
)

async def main():
    result = await Runner.run(agent, "罗马帝国什么时候灭亡？")
    print(result.final_output)

if __name__ == "__main__":
    asyncio.run(main())
  1. Agent(name, instructions, ...):name 用于追踪与 handoff;instructions 是系统提示(也可传一个函数,根据上下文动态生成)。其他常用参数:model、tools、handoffs、output_type(结构化最终输出)、model_settings(温度/effort 等)、input_guardrails/output_guardrails。
  2. Runner.run(agent, input) 是 async,返回 RunResult。同步场景用 Runner.run_sync(...);流式用 Runner.run_streamed(...)(4.9)。
  3. result.final_output 是 agent 的最终回答(默认是字符串;设了 output_type 则是该类型的实例)。

4.3 工具:@function_tool

tool_agent.pypython
from agents import Agent, Runner, function_tool, RunContextWrapper

@function_tool
async def fetch_weather(city: str) -> str:
    """查询某城市当前天气。

    Args:
        city: 城市名。
    """
    return f"{city}：晴，24°C"

@function_tool(failure_error_function=lambda ctx, err: "查询失败，请稍后再试。")
def get_profile(user_id: str) -> str:
    """按用户 ID 查资料。"""
    if user_id == "u_1":
        return "..."
    raise ValueError("not found")

agent = Agent(
    name="Assistant",
    instructions="需要时调用工具。",
    tools=[fetch_weather, get_profile],
)
  1. @function_tool 把普通函数变工具:名字取自函数名(可 name_override),描述取自 docstring,参数 schema 由类型注解自动生成(用 Pydantic/TypedDict 表达复杂结构)。同步异步皆可。这等于 Anthropic 的 @beta_tool。
  2. RunContextWrapper:把它作为第一个参数(ctx: RunContextWrapper[T]),工具就能拿到你传给 Runner.run(..., context=...) 的依赖对象(DB 连接、用户身份等)。这个 context 不进模型上下文,纯本地依赖注入。
  3. failure_error_function:工具抛异常时,用它生成回给模型的错误文本(让 agent 自愈);传 None 则异常上抛由你处理。默认会记录错误并回一段通用提示。
  4. 安全同理:工具体内的路径/命令仍是不可信输入(模型决定参数),该做的校验、隔离、白名单一个都不能少——框架不替你做这件事。

4.4 运行与 RunResult

方法说明
await Runner.run(agent, input)异步,返回 RunResult
Runner.run_sync(agent, input)同步封装(内部跑事件循环)
Runner.run_streamed(agent, input)返回 RunResultStreaming,配 .stream_events()(4.9)

RunResult 关键字段:

  • final_output — 最终输出(字符串或 output_type 实例)。
  • new_items — 本次运行产生的所有项(消息、工具调用、工具结果、handoff…)。
  • to_input_list() — 把本次结果转成「下一轮的 input」,用于手动续接多轮(见下)。
  • last_agent — 最终是哪个 agent 收尾的(handoff 后很有用)。
手动多轮：to_input_list()python
result = await Runner.run(agent, "金门大桥在哪个城市？")
print(result.final_output)

new_input = result.to_input_list() + [{"role": "user", "content": "那在哪个州？"}]
result = await Runner.run(agent, new_input)        # 带上一轮全部上下文
print(result.final_output)

异常:MaxTurnsExceeded(超 max_turns)、ModelBehaviorError(模型输出畸形)、InputGuardrailTripwireTriggered/OutputGuardrailTripwireTriggered(护栏触发,见 4.7)。

4.5 托管工具与 agent.as_tool

内置的 OpenAI 托管工具被封装成类,直接塞进 tools:

hosted_tools.pypython
from agents import Agent, Runner, WebSearchTool, FileSearchTool

agent = Agent(
    name="Assistant",
    tools=[
        WebSearchTool(),
        FileSearchTool(max_num_results=3, vector_store_ids=["VS_ID"]),
        # CodeInterpreterTool() / ImageGenerationTool() / ComputerTool(...) / HostedMCPTool(...)
    ],
)

还能把一个 Agent 当工具给另一个 Agent 用(agent-as-tool,适合「编排者调用专家但不交出控制权」):

agent_as_tool.pypython
spanish = Agent(name="Spanish agent", instructions="把用户消息翻成西班牙语")

orchestrator = Agent(
    name="orchestrator",
    instructions="你是翻译协调者，用给你的工具完成翻译。",
    tools=[
        spanish.as_tool(
            tool_name="translate_to_spanish",
            tool_description="把用户消息翻成西班牙语",
        ),
    ],
)

as_tool vs handoff:as_tool 是「调用专家拿结果,控制权仍在协调者」(像函数调用);handoff 是「把整段对话移交给另一个 agent,由它接管收尾」(下一节)。

4.6 Handoffs:多智能体路由

handoffs.pypython
history = Agent(name="History Tutor",
                handoff_description="历史问题专家",
                instructions="清晰简洁地回答历史问题。")
math = Agent(name="Math Tutor",
             handoff_description="数学问题专家",
             instructions="一步步讲数学并给例子。")

triage = Agent(
    name="Triage Agent",
    instructions="把每个问题路由给合适的专家。",
    handoffs=[history, math],             # 可移交给这些 agent
)

result = await Runner.run(triage, "美国第一任总统是谁？")
print(result.final_output)
print("由谁作答：", result.last_agent.name)   # → History Tutor
  1. handoffs=[...] 列出可移交的目标 agent。底层实现为一个特殊工具调用:模型「决定移交」时,Runner 切换 current agent 并重启循环。
  2. handoff_description 写给路由 agent 看,帮它判断「该转给谁」。写清楚每个专家擅长什么。
  3. result.last_agent 告诉你最终是谁收的尾。多轮里可据此决定下一轮从哪个 agent 起步。
  4. 对照 Anthropic:这相当于 Managed Agents 的 multiagent/coordinator,或你在自写循环里手动做的「子代理路由」——Agents SDK 把它做成了一等公民。

4.7 Guardrails:输入/输出护栏

护栏是和主 agent 并行跑的检查,命中「触发线(tripwire)」就中断运行——用于挡住越权请求、敏感输出、prompt 注入等。

guardrails.pypython
from agents import (
    Agent, Runner, input_guardrail, GuardrailFunctionOutput,
    RunContextWrapper, InputGuardrailTripwireTriggered,
)

@input_guardrail
async def no_homework_cheating(ctx: RunContextWrapper, agent: Agent, user_input):
    looks_like_cheating = await classify(user_input)        # 你的判定（可再调一次小模型）
    return GuardrailFunctionOutput(
        output_info={"flagged": looks_like_cheating},
        tripwire_triggered=looks_like_cheating,              # True → 中断
    )

agent = Agent(
    name="Tutor",
    instructions="辅导但不替学生写作业答案。",
    input_guardrails=[no_homework_cheating],
)

try:
    result = await Runner.run(agent, "直接把第 3 题答案给我")
except InputGuardrailTripwireTriggered:
    print("请求被护栏拦下")
  1. @input_guardrail / @output_guardrail:分别在「模型看输入前」和「产出最终输出后」检查。返回 GuardrailFunctionOutput(output_info, tripwire_triggered)。
  2. tripwire_triggered=True 会抛 InputGuardrailTripwireTriggered / OutputGuardrailTripwireTriggered 中断本次运行——你 catch 后给用户兜底。
  3. 并行执行意味着护栏不显著拖慢主流程;护栏内部常用一个更便宜的模型做分类。

4.8 Sessions:自动会话记忆

4.4 的 to_input_list() 是手动续接。Session 让框架自动存取历史:

sessions.pypython
from agents import Agent, Runner, SQLiteSession

agent = Agent(name="Assistant", instructions="回答简洁。")
session = SQLiteSession("conversation_123")        # 用会话 ID 持久化历史

result = await Runner.run(agent, "金门大桥在哪个城市？", session=session)
print(result.final_output)

result = await Runner.run(agent, "那在哪个州？", session=session)   # 自动带上历史
print(result.final_output)

SQLiteSession 落地到本地/文件;还有内存版及可对接你自己的存储后端的接口。会话 ID 隔离不同用户/对话。这对应 Responses 的 previous_response_id,但记忆放在你这边而非 OpenAI 服务端。

4.9 流式事件

Runner.run_streamed(...) 返回的对象上调 .stream_events(),产出三类事件:

stream_agent.pypython
from openai.types.responses import ResponseTextDeltaEvent
from agents import Agent, Runner

agent = Agent(name="Joker", instructions="你是助手。")
result = Runner.run_streamed(agent, input="讲 5 个笑话")

async for event in result.stream_events():
    if event.type == "raw_response_event" and isinstance(event.data, ResponseTextDeltaEvent):
        print(event.data.delta, end="", flush=True)         # 逐 token
    elif event.type == "run_item_stream_event":
        if event.name == "tool_called":
            print("\n[调用工具]", event.item)
        elif event.name == "tool_output":
            print("\n[工具结果]", event.item)
    elif event.type == "agent_updated_stream_event":
        print("\n[切换到 agent]", event.new_agent.name)      # handoff 时
  1. raw_response_event 是透传底层 Responses 的语义事件(就是 3.9 那些 response.*)。要逐 token 流文本,就匹配 ResponseTextDeltaEvent 取 event.data.delta。
  2. run_item_stream_event 是高层语义:event.name ∈ message_output_created / tool_called / tool_output / handoff_requested / handoff_occured / reasoning_item_created 等,event.item 是对应的项。用它做 UI 上「正在调用 X 工具」这类提示最合适。
  3. agent_updated_stream_event:当前 agent 变了(handoff),event.new_agent 是新 agent。
  4. 对照 Anthropic:这就是一层封装——raw_response_event ≈ 原始 SSE 事件,run_item_stream_event ≈ 你自己在循环里根据内容块类型打的「语义日志」,只是 SDK 替你做了。

4.10 Tracing:开箱即用的可观测性

Agents SDK 默认自动追踪每次运行:LLM 调用、工具调用、handoff、护栏都记成 span,可在 OpenAI 后台的 Traces 面板里看整条轨迹。这对调多智能体系统极有用——你能看清「为什么路由到了那个 agent」「哪个工具慢」。

  • 默认开启;可用 RunConfig(tracing_disabled=True) 关闭,或用 trace(...) 上下文把多次 run 归到同一条 trace。
  • 可接第三方可观测平台(通过 trace processor 导出)。
  • 敏感数据合规时可配置不上传具体输入输出,只留结构。

Anthropic 侧没有等价的内建追踪(Managed Agents 有 observability/事件流);自托管循环要可观测得自己埋点。Agents SDK 的 tracing 是它相对「手搓循环」的一大卖点。

6同一个 Agent,三套写法对照 CORE

把前面拆开讲的东西合起来:同一个带工具的 agentic loop,分别用 Chat / Responses / Agents SDK 写一遍。任务和工具都一样——一个 get_weather(city) 工具,问「北京和上海哪个更热」(模型需要并行/连续调两次)。盯住三段代码谁在管循环、谁在管历史、工具形状有何不同。把 get_weather 换成你真实的 read_file/run_bash 一套(参见本站 Anthropic 教程的 miniclaude),骨架一字不变。

写法 A — Chat Completions(手写循环,无状态)

agent_chat.pypython
import json
from openai import OpenAI
client = OpenAI()

def get_weather(city: str) -> str:
    return f"{city}：晴 30°C" if city == "北京" else f"{city}：多云 27°C"

tools = [{
    "type": "function",
    "function": {                                  # Chat：schema 嵌在 "function" 下
        "name": "get_weather",
        "description": "查询某城市当前天气",
        "parameters": {"type": "object",
                       "properties": {"city": {"type": "string"}},
                       "required": ["city"], "additionalProperties": False},
        "strict": True,
    },
}]

messages = [{"role": "user", "content": "北京和上海哪个更热？"}]
while True:
    resp = client.chat.completions.create(model="gpt-5.5", messages=messages, tools=tools)
    msg = resp.choices[0].message
    messages.append(msg)                           # 把整条助手消息追加回历史（无状态：每轮全量重发）
    if not msg.tool_calls:
        print(msg.content); break
    for call in msg.tool_calls:                     # 可能并行两次
        args = json.loads(call.function.arguments)  # arguments 是 JSON 字符串
        messages.append({"role": "tool", "tool_call_id": call.id,
                         "content": get_weather(args["city"])})   # 每个结果一条 tool 消息

写法 B — Responses(手写循环,有状态)

agent_responses.pypython
import json

client = OpenAI()

def get_weather(city: str) -> str: ...

tools = [{
    "type": "function",                            # Responses：扁平，name/parameters 与 type 同级
    "name": "get_weather",
    "description": "查询某城市当前天气",
    "parameters": {"type": "object",
                   "properties": {"city": {"type": "string"}},
                   "required": ["city"], "additionalProperties": False},
    "strict": True,
}]

resp = client.responses.create(model="gpt-5.5", input="北京和上海哪个更热？", tools=tools)
while True:
    calls = [it for it in resp.output if it.type == "function_call"]
    if not calls:
        print(resp.output_text); break
    outputs = [{
        "type": "function_call_output",            # Responses：用 function_call_output 回填
        "call_id": c.call_id,
        "output": get_weather(json.loads(c.arguments)["city"]),
    } for c in calls]
    resp = client.responses.create(
        model="gpt-5.5",
        previous_response_id=resp.id,              # 有状态：只发工具结果，不重发历史
        input=outputs, tools=tools,
    )

agent_sdk.pypython
import asyncio
from agents import Agent, Runner, function_tool

@function_tool
def get_weather(city: str) -> str:
    """查询某城市当前天气。"""
    return f"{city}：晴 30°C" if city == "北京" else f"{city}：多云 27°C"

agent = Agent(name="Weather", instructions="需要时查天气再回答。", tools=[get_weather])

async def main():
    result = await Runner.run(agent, "北京和上海哪个更热？")   # 循环、回填、并行全由 Runner 管
    print(result.final_output)

asyncio.run(main())
  1. 循环归属:A/B 你自己写 while(换来对每步的拦截/审批/日志能力);C 由 Runner 托管(换来少写代码 + 内建追踪)。这正是本站 Anthropic 教程里「手写 loop vs Tool Runner」的同一权衡。
  2. 历史归属:A 无状态,每轮把全量 messages 重发;B 用 previous_response_id 让服务端记历史,只发增量;C 框架替你管(默认每次 run 独立,跨轮用 Session 或 to_input_list())。
  3. 工具形状差异(搬代码最容易错):A 的 schema 嵌在 "function" 下、结果回填用 role:"tool" 消息;B 扁平、回填用 function_call_output item;C 用 @function_tool 从签名/docstring 自动生成 schema,你完全不碰 JSON。
  4. arguments 都是 JSON 字符串(A/B 都要 json.loads);C 帮你解析好直接当函数参数传入。

7错误处理与重试

和 Anthropic 一样:用 SDK 的类型化异常,从具体到一般链式捕获,区分「可重试」(429/5xx/网络)和「不可重试」(4xx)。别用字符串匹配错误信息。

HTTPopenai.* 异常类可重试常见原因
400BadRequestError否参数/形状错误(如工具 schema 写错)
401AuthenticationError否API key 无效/缺失
403PermissionDeniedError否无权访问该模型/功能
404NotFoundError否模型 ID 写错、端点不存在
409ConflictError是(SDK 会重试)资源冲突
422UnprocessableEntityError否请求体语义错误
429RateLimitError是RPM/TPM 超限
≥500InternalServerError是服务端错误
网络APIConnectionError(含 APITimeoutError)是连接失败/超时
基类APIStatusError(非 2xx)/ APIError—兜底
errors.py — 分层捕获python
import openai

try:
    resp = client.responses.create(model="gpt-5.5", input="...")
except openai.NotFoundError:                 # 404：多半是 model ID 写错
    raise
except openai.RateLimitError as e:           # 429：SDK 已自动退避重试；到这说明重试也耗尽
    retry_after = e.response.headers.get("retry-after")
    ...
except openai.APIStatusError as e:           # 其他非 2xx
    print(e.status_code, e.request_id, e.message)
    if e.status_code >= 500:
        ...                                  # 服务端错误，可重试
    else:
        raise                                # 4xx，不该重试
except openai.APITimeoutError:               # 是 APIConnectionError 的子类，放它前面
    ...
except openai.APIConnectionError:            # 其他网络失败
    ...
  1. 顺序从具体到一般。APITimeoutError 是 APIConnectionError 的子类,要放前面;APIStatusError 是所有「拿到了非 2xx 响应」的基类。
  2. 每个异常带诊断信息:e.status_code、e.request_id(报障时给 OpenAI)、e.message、e.code/e.type(细分错误类型)、e.response.headers(限流头)。

SDK 自动重试与超时

retry_config.pypython
from openai import OpenAI
import httpx

client = OpenAI(
    max_retries=4,        # 默认 2；自动重试 408/409/429/≥500 + 连接错误，指数退避
    timeout=60.0,         # 单位：秒（可传 httpx.Timeout 做更细粒度控制）
)

# 也可对单次请求覆盖（不改全局客户端）
resp = client.with_options(max_retries=5, timeout=20.0).responses.create(...)
  1. 默认就会重试。SDK 对 408/409/429/≥500 和连接错误自动指数退避重试,默认 max_retries=2。只有需要超出默认行为时才手写重试。
  2. 限流(429)看 retry-after。SDK 会读这个头自动等待;你自己处理时也按它退避。响应头还有 x-ratelimit-remaining-requests/...-tokens 可用于预算。
  3. 超时单位是秒(对照 Anthropic Python 也是秒、TS 是毫秒)。高 max_tokens 用流式避免超时。

流式里的错误 & Agents SDK 的异常

  • 流式:错误可能在流中途以 error 事件出现(如 InternalServerError/限流),用 with/迭代时会以上述异常抛出,照样分层捕获。
  • Agents SDK 在这些底层异常之上还有自己的一层:MaxTurnsExceeded(超 max_turns)、ModelBehaviorError(模型输出畸形)、InputGuardrailTripwireTriggered/OutputGuardrailTripwireTriggered(护栏触发,见 4.7)。底层的 RateLimitError 等仍会照常冒上来。

对照 Anthropic:异常体系几乎一一对应(RateLimitError/NotFoundError/APIConnectionError/APIStatusError 同名同义),迁移时把 anthropic. 换成 openai. 即可,链式捕获的结构不变。

8最佳实践 & 速查

选型

  • 新项目默认 Responses;要多智能体/护栏/追踪上 Agents SDK;只为兼容性/迁移老代码用 Chat Completions。
  • 三者可混用:Agents SDK 底层就是 Responses;Responses 与 Chat 的工具/结构化心智一致,差在形状。

函数调用(三套通用)

  • 把整条助手消息/输出追加回历史,再处理工具——否则模型「失忆」。
  • arguments 永远是 JSON 字符串,必须 json.loads;流式时按 index 累积碎片再解析。
  • 结果回填要对上 ID(tool_call_id / call_id);失败也要回结果(错误文本),让模型自愈。
  • 工具形状别搞混:Chat 嵌在 function 下,Responses 扁平。

结构化输出

  • 用 strict:True + additionalProperties:False + 全字段 required(可选字段用 null 联合类型表达)。
  • 优先 .parse() + Pydantic;别用只保证「是 JSON」的 json_object。
  • 判 finish_reason/status:被拒或截断时输出可能不符合 schema。

流式

  • 高 max_tokens/max_completion_tokens 时一律流式,避免 HTTP 超时。
  • Responses 的 response.completed(及 Chat 的累积、Agents 的 stream 收尾)给你完整终态,等价 Anthropic 的 get_final_message()。
  • Chat 流式要 usage,记得 stream_options={"include_usage": True};choices 可能为空先判空。
  • 对未知事件类型优雅忽略(前向兼容)。

推理模型

  • 用 reasoning.effort 控深度;要可读摘要设 summary;原始思维链不返回。
  • 多轮保留 reasoning item(previous_response_id 自动保留;手动历史别丢)。
  • 推理模型用 max_completion_tokens,别用 max_tokens;很多推理模型不接受采样参数。

Agents SDK

  • @function_tool 自动生成 schema;用 RunContextWrapper 注入本地依赖(不进模型上下文)。
  • 多轮:简单用 Session 自动记忆,精细用 to_input_list() 手动续。
  • as_tool(保留控制权)vs handoff(移交收尾)按需选。
  • 护栏用更便宜的模型并行跑;max_turns 防失控;善用内建 Tracing 调多智能体。
  • 工具体内仍要做安全校验/隔离——框架不替你做。
