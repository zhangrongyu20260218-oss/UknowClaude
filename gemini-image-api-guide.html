Google Gemini 图像生成 API 完整讲解

Gemini 媒体生成：原生图像（Nano Banana）、Imagen，以及新的 Gemini Omni（任意输入→视频）。以官方 google-genai SDK 为主线，cURL/REST 对照。

时效与来源说明（务必先读）。本文核对日期 2026-06-30。事实来自 ai.google.dev 官方页的搜索摘要、官方 googleapis/python-genai SDK 与 cookbook。编写时本环境出口代理封锁了对 ai.google.dev 的直接抓取，故凡涉及模型 ID 的 -preview 后缀、定价/token 折算、分辨率档位、字段默认值等易变项，均以 ⚠️ 标注，请以官方文档与控制台为准。

1概览与 2026 格局

截至 2026 年中，Google 图像生成格局发生了重大变化：Gemini 原生图像生成（"Nano Banana" 系列）已成为官方主推，通过 generateContent 返回图片；而专用文生图模型 Imagen 4 全系列已弃用，计划 2026-08-17 关停，官方建议迁移到 Nano Banana。

路线是什么状态本文
Gemini 原生（Nano Banana）多模态模型，generateContent 直接返回图片，支持图文混排、编辑、对话修图✅ 官方主推、活跃第 2–4 节
Imagen 已弃用专用文生图，:predict / generate_images⚠️ 2026-08-17 关停第 5 节（仅迁移参考）
Gemini Omni preview · 新任意输入（文/图/音/视频）→ 视频输出（含同步音频），走 Interactions API2026 I/O 发布，逐步开放第 7 节

判别口诀：图片在请求里（输入）= 图像理解；图片在响应里（输出）= 图像生成。本文只讲"生成"。把图片喂给模型做分析/OCR/问答属于"理解"，走文本理解计费——见 第 7 节 与 文本篇·多模态输入。

新文档里图像生成页有时归在较新的 "Interactions API" 标题下（取图字段为 interaction.output_image.data）。本文以更通用、SDK 稳定的 generate_content + inline_data 写法为主；若你用 Interactions API，取图字段以该页为准。

2Gemini 原生图像（Nano Banana）

2.1 模型 ID

别名精确模型 ID ⚠️定位
Nano Banana Progemini-3-pro-image-preview最高质量、推理驱动、专业级生成/编辑
Nano Banana 2gemini-3.1-flash-image高吞吐、高效率、低价位
Nano Banana（初代）gemini-2.5-flash-image速度/效率优先（较早一代）

⚠️ Gemini 3 系列图像模型多标为 preview；-preview 后缀会随转正而变。调用前以官方 Models 页 为准，或 GET /v1beta/models 查询。

2.2 调用方式：generateContent + responseModalities

核心思路：调用普通的 generateContent，把图像模型放进去；可选用 responseModalities（SDK：response_modalities）声明要 TEXT、IMAGE 或两者。返回的 parts 里凡 inlineData（SDK：inline_data）即图片，含 mimeType 与 base64 data。

generate
from google import genai

client = genai.Client()   # 读取 GEMINI_API_KEY

resp = client.models.generate_content(
    model="gemini-3.1-flash-image",        # 或 gemini-3-pro-image-preview
    contents=["画一道以 Gemini 为主题、摆盘精致的香蕉料理"],
)

for part in resp.parts:
    if part.text is not None:
        print(part.text)
    elif part.inline_data is not None:
        part.as_image().save("generated_image.png")   # SDK 便捷方法转 PIL.Image
from google.genai import types

resp = client.models.generate_content(
    model="gemini-3.1-flash-image",
    contents="一张关于会飞的运动鞋的卡通信息图",
    config=types.GenerateContentConfig(
        response_modalities=["IMAGE"],
        image_config=types.ImageConfig(aspect_ratio="9:16"),
    ),
)
for part in resp.parts:
    if part.inline_data:
        part.as_image().save("out.png")
REST
curl -s "https://generativelanguage.googleapis.com/v1beta/models/gemini-3.1-flash-image:generateContent" \
  -H "x-goog-api-key: $GEMINI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "contents": [{ "parts": [{ "text": "一张关于会飞的运动鞋的卡通信息图" }] }],
    "generationConfig": { "responseModalities": ["TEXT", "IMAGE"] }
  }' > resp.json

# 取出第一段 inlineData 的 base64 并解码写盘
jq -r '.candidates[0].content.parts[] | select(.inlineData) | .inlineData.data' resp.json \
  | base64 --decode > generated_image.png

字段路径：candidates[].content.parts[].inlineData.{mimeType,data}，data 为 base64。⚠️ 经典 REST 用 camelCase 的 inlineData；若走 Interactions API 端点，取图字段为 output_image.data，请求/响应结构不同。

3图像编辑 / 多图输入 / 对话式修图

这是 Nano Banana 相对 Imagen 的核心优势：因为它本质是多模态对话模型。

  • 图像编辑 / 多图输入：把一张或多张图片连同文本一起放进 contents，模型可做编辑、风格迁移、合成。输入图片用 inline_data（base64 + mimeType）或 File API 传入。
  • 对话式修图（推荐）：用 client.chats.create(...) 建多轮会话，逐轮自然语言下指令（"把背景换成日落"）。官方说明多轮会话是迭代图像的推荐方式，并依赖 Thought Signatures 在轮次间保留视觉上下文。
  • 多图上限：单次请求最多支持约 3,600 个图像文件。⚠️
edit / chat
from google.genai import types
from PIL import Image

input_image = Image.open("room.png")

chat = client.chats.create(
    model="gemini-3.1-flash-image",
    config=types.GenerateContentConfig(response_modalities=["TEXT", "IMAGE"]),
)
resp = chat.send_message(["把背景换成日落", input_image])
for part in resp.parts:
    if part.inline_data:
        part.as_image().save("edited.png")
# 继续 chat.send_message("再把灯打开") 逐步迭代——上下文自动保留

5.3 示例

Imagen
from google import genai
from google.genai import types

client = genai.Client()
resp = client.models.generate_images(
    model="imagen-4.0-generate-001",
    prompt="前景一把雨伞，背景是雨夜的天空",
    config=types.GenerateImagesConfig(
        number_of_images=1,
        aspect_ratio="3:4",
        person_generation="allow_adult",
        output_mime_type="image/jpeg",
    ),
)
resp.generated_images[0].image.save("imagen_out.jpg")
curl -X POST \
  "https://generativelanguage.googleapis.com/v1beta/models/imagen-4.0-generate-001:predict" \
  -H "x-goog-api-key: $GEMINI_API_KEY" -H "Content-Type: application/json" -d '{
    "instances": [ { "prompt": "拿着红色滑板的机器人" } ],
    "parameters": { "sampleCount": 4, "aspectRatio": "16:9", "personGeneration": "allow_adult" }
  }' > imagen_resp.json

# 取第一张图的 base64 并存盘（字段名以官方为准，常见为 bytesBase64Encoded）
jq -r '.predictions[0].bytesBase64Encoded' imagen_resp.json | base64 --decode > imagen_out.png

⚠️ 旧的 import google.generativeai as genai + ImageGenerationModel(...) 属老 SDK，新代码统一用 from google import genai。:predict 响应取图字段名（bytesBase64Encoded vs 其它）、outputMimeType 默认值需以官方页核对。所有 Imagen 生成图同样带 SynthID 水印。

选型建议：

7Gemini Omni（任意输入 → 视频）preview · 新

本节可信度分级，请务必看清。Gemini Omni 属新发布 / preview，走新的 Interactions API（client.interactions.create，与经典 generateContent 不同端点，不在 v1beta discovery doc 中）。官方 Omni 页 ai.google.dev/gemini-api/docs/omni 在本环境被组织出口策略封锁（403）无法直连核对。分级如下：

  • ✅ 已由官方 SDK 源码核实：client.interactions.create(model, input, ...)、多轮承接字段 previous_interaction_id、多模态输出约定 response_modalities=[...]。
  • ⚠️ 未获官方证实：Omni 视频的具体输出字段与参数形态。第三方教程流传的 response_format={"type":"video","aspect_ratio":...} + interaction.output_video.data 与官方 response_modalities 约定相冲突，请勿照抄，以官方 Omni 页为准。
  • ℹ️ 官方 SDK README 按名文档化的视频生成是 Veo（client.models.generate_videos()），并未收录 Omni 视频接口——见 7.2 末的对照。

7.1 它是什么

Gemini Omni Flash 是"任意输入 → 视频"的多模态生成模型：把文本 / 图像 / 音频 / 视频任意组合作为输入，产出带同步音频的视频，且视频内容锚定 Gemini 对真实世界的知识（物理、地理、文化、语言）。发布时约 10 秒短片；官方称未来会扩展到图像、音频等输出模态。

项值
模型 IDgemini-omni-flash-preview ⚠️（来自搜索，未从 SDK/discovery 核对）
API 表面✅ Interactions API：client.interactions.create(...)（非 generateContent）
输入文本 + 图像 + 音频 + 视频（任意组合）⚠️
输出视频（含同步音频）；确切读取字段未证实（见 7.2，勿用第三方教程的 output_video.data）

7.2 生成一段视频（调用骨架）

下面用官方 SDK 已核实的 interactions.create 约定套用到 Omni 模型。注意：官方 SDK README 的多模态输出用 response_modalities（例如生成图像是 response_modalities=['IMAGE']），故视频推测为 ['VIDEO']；但 README 未按名收录 Omni 视频，输出字段名未证实，以官方 Omni 页为准。

Interactions API
from google import genai

client = genai.Client()

# interactions.create / input / response_modalities 均为官方 README 已核实的约定
interaction = client.interactions.create(
    model="gemini-omni-flash-preview",   # ⚠️ 模型 ID 来自搜索，未从 SDK/discovery 核对
    input="霓虹灯下、飞行汽车穿梭的赛博朋克未来城市，连续平滑运镜",
    response_modalities=["VIDEO"],           # ⚠️ 'VIDEO' 系按 SDK 约定推测（图像为 'IMAGE'）
)

# ⚠️ 视频输出的确切读取方式未证实：REST 下产物在 steps 数组里，
#    SDK 可能有便捷字段。以官方 Omni 页为准，别照抄第三方教程的 output_video.data。
print(interaction)

不要照抄第三方教程的这套写法：response_format={"type":"video","aspect_ratio":"16:9"} + interaction.output_video.data。它与官方 SDK 的 response_modalities 约定相冲突、未获一手源证实，很可能不对。时长 / 分辨率 / FPS / 画幅是否可在调用层配置也未证实。

要现在就用稳定的视频生成？看 Veo。官方 SDK README 按名文档化的视频生成是 Veo：client.models.generate_videos()，支持文本→视频、图像→视频、视频→视频。Omni 的差异化在于"任意模态输入 + 对话式多轮编辑 + 锚定真实世界知识"，但目前 preview、接口未在 SDK README 定型。生产选型时把两者都纳入评估。

7.3 对话式多轮编辑（Omni 的招牌能力）

Omni 的核心卖点是通过 Interactions API 做对话式迭代编辑：用自然语言描述"改什么、保留什么"，每次编辑建立在上一次之上，跨镜头保持角色一致、场景/地点延续，无需从头重生成。Interactions API 在服务端维护状态，续编辑靠回传 previous_interaction_id（= 上一次的 interaction.id）承接上下文——该字段已由官方 SDK README 核实：

多轮承接
first = client.interactions.create(model="gemini-omni-flash-preview",
                                   input="一段海边日出的镜头",
                                   response_modalities=["VIDEO"])

# 承接上一次，做增量编辑（服务端保留视觉上下文）
edited = client.interactions.create(
    model="gemini-omni-flash-preview",
    input="把天空换成暴风雨，其余保持不变",
    previous_interaction_id=first.id,     # ✅ 官方 README 已核实
    response_modalities=["VIDEO"],
)

与 Nano Banana 的"对话式修图"（图像）是同一思路的视频版：都靠会话状态保留视觉上下文，逐步迭代而非每次从零生成。

7.4 选型定位

  • 要视频（尤其任意模态输入、带同步音频、可对话式剪辑）→ Gemini Omni。
  • 要静态图（生成/编辑/合成）→ Nano Banana（第 2–4 节）。
  • Omni 仍在 preview 且逐步开放；生产前确认你的项目已获访问权限与配额。

8图像理解 ≠ 图像生成

很多人把"给模型一张图让它分析/描述/OCR"误当成图像生成，二者完全不同：

  • 图像理解＝多模态文本理解：图片作为输入传给 Gemini（inline_data 或 File API），模型输出文本（描述、问答、检测、OCR）。消耗输入 token，走文本理解计费，不产生图片。
  • 图像生成（本文 2–5 节）＝模型输出图片。

两者甚至可在同一次 generateContent 里组合：输入一张图、输出一张改后的图 + 一段说明文本。图像理解的写法见 文本篇 · 第 12 节 多模态输入。
