用 Claude Agent SDK 造一个真实应用

写给会 HTTP/SDK、会工程，但还没把 LLM 当成「带几处反直觉行为的运行时」来用的资深程序员。我们不堆 API 列表，而是从零造一个能跑的 CLI 编码 Agent —— miniclaude，逐行解释每一行为什么这么写。主线 Python 官方 SDK，默认模型 claude-opus-4-8。

一个 Agent 在工程上只有一句话：把模型放进一个循环里，循环每一圈让模型选工具、你执行工具、把结果喂回去，直到模型说「做完了」。本文把这句话拆成可运行代码：工具怎么设计才安全、循环怎么写才不丢 tool_use、长任务上下文怎么不爆、跨会话怎么记住、出错/被拒/要人审怎么办。每一条都对应 SDK 的真实行为，不是凭印象。

我们要造的东西。miniclaude 是一个终端里的自治编码助手：你给它一句任务（「把 utils.py 里的同步函数改成 async」），它自己读文件、改文件、跑命令、必要时联网搜索，边想边干，干完汇报。它是一个「迷你版 Claude Code」，足够覆盖 Agent 的全部核心能力，又小到能逐行读完。

关于「Agent SDK」这个词。构建 Agent 有两条官方路径：(A) 自己托管算力—— 用 Anthropic 官方 SDK（anthropic）在 /v1/messages 上自己写 agentic loop；(B) 托管 Agent（Managed Agents）—— Anthropic 替你跑循环、并提供每个会话一个沙箱容器。本文走 (A)：你掌控每一次工具执行、审批、日志，最适合「装进自己产品里」。第 14 节会指出何时该换成 (B)。

1心智模型：Agent 到底是什么

先把名词去魅。模型 API 本身是无状态的：一次 messages.create 进去一段对话历史，出来一段回复。它不会「记得」上一次，也不会自己执行任何东西。所谓 Agent，就是你在它外面套的一个控制循环：

agentic loop 的本质（伪代码）pseudocode
messages = [{"role": "user", "content": 任务}]
while True:
    resp = model(messages, tools)          # 模型看历史 + 可用工具，决定下一步
    if resp.stop_reason == "end_turn":     # 模型说「我说完了，不需要工具」
        break
    results = [执行(tu) for tu in resp 里的 tool_use 块]   # 你来跑工具
    messages += [resp, {"role":"user", "content": results}]  # 把结果喂回去
return resp 里的最后一段文本

三件事决定了这个循环的全部复杂度，也是本文的主干：

  • 工具（tools）：模型能调用什么。工具的「形状」决定了你的程序能对每个动作做什么 —— 拦截、审批、渲染、并行（第 3、4 节）。
  • 停止条件（stop_reason）：什么时候退出循环。end_turn / tool_use / pause_turn / refusal / max_tokens 每一种都要处理（第 5、8、13 节）。
  • 上下文（context）：历史会越滚越长。长跑 Agent 必须主动管理上下文，否则要么爆 1M 窗口，要么烧钱（第 9、10、11 节）。

什么时候不该造 Agent

Agent 是最贵、最慢、最难调的那一层。官方给的判据是四个都为「是」才上：

判据含义
复杂度任务多步、无法提前完全写死流程（「把设计文档变成 PR」而不是「抽取 PDF 标题」）
价值结果值得更高的成本与延迟
可行性模型确实擅长这类任务
错误成本错了能被发现并回滚（有测试、有 review）

任何一个为「否」，退回更简单的层：单次调用，或你自己编排的 workflow（用 tool use 但循环逻辑由你的代码主导）。编码助手四个都满足，所以我们造它。

2项目骨架与客户端

先把工程搭起来。三个文件：tools.py（工具定义+执行器）、agent.py（循环）、main.py（入口）。本节先建客户端，并把传输层的反直觉点一次讲清。

client.pypython
import anthropic

MODEL = "claude-opus-4-8"

client = anthropic.Anthropic(
    max_retries=4,
    timeout=600.0,
)
  1. 第 1 行 官方 SDK 包名就是 anthropic。不要在 Python 项目里手搓 requests 去打 /v1/messages——SDK 帮你处理重试、超时缩放、流式累积、类型，全丢掉得不偿失。
  2. 第 3 行 模型 ID 是完整即用的字符串 claude-opus-4-8，不要自己补日期后缀（...-20251101 之类会 404）。Agent 默认就用 Opus 4.8：它是长跑自治任务最强的一档，别为省钱擅自降级——那是用户的决定。
  3. 第 5 行 不传 api_key 是推荐写法：SDK 会按顺序从环境解析凭据（ANTHROPIC_API_KEY → ANTHROPIC_AUTH_TOKEN → ant auth login 的本地 profile）。永远不要把 key 硬编码进源码。
  4. 第 6 行 max_retries 默认 2，SDK 自动重试 408/409/429/5xx + 连接错误，指数退避。把它调到 4 让长跑 Agent 更稳。注意：超时也会被重试，最坏挂钟时间 ≈ timeout × (max_retries+1)。
  5. 第 7 行 默认超时 10 分钟。Opus 4.8 在 effort=high 下单个请求可能跑好几分钟；timeout 单位是秒（TypeScript SDK 是毫秒，别记混）。

反直觉点：高 max_tokens 必须流式。非流式请求在 max_tokens 较大时会撞上 SDK 的 HTTP 超时保护——SDK 估算请求会超过 ~10 分钟时直接抛 ValueError。规则：输出可能 > ~16K token 时一律用 messages.stream()（第 7 节）。Opus 4.8 支持最高 128K 输出，但 128K 这种量级强制要流式。

异步与高并发

单个交互式 Agent 用同步客户端足矣。如果你要同时跑很多 Agent（比如服务端为多个用户各起一个），用 AsyncAnthropic，并安装 anthropic[aiohttp] 切到 aiohttp 后端以降低高并发开销。本文为可读性用同步版。

3工具面设计与安全边界

这一节没有代码，但它是 Agent 工程里最重要的设计决策，比循环本身更难。核心问题：同一个动作，是给模型一个 bash 工具，还是给它一个专用工具？

模型只会「发出工具调用」，真正执行的是你的 harness。所以工具的形状 = 你的 harness 能对这个动作做什么。

bash 工具专用工具（如 edit_file）
给模型的能力极广，几乎能做任何事受限于你定义的 schema
给 harness 的把手只有一个不透明的命令字符串，每个动作形状都一样带类型的参数，可拦截/校验/渲染/审计
能否安全网关难：bash -c "curl -X POST ..." 没法识别易：send_email 工具天然可挂审批
能否做新鲜度校验不能能：edit 可拒绝「文件自上次读取后被改过」的写入
能否并行调度不能：分不清并行安全的 grep 和不安全的 git push能：只读工具可标记为并行安全

经验法则：用 bash 起步拿广度；当某个动作需要网关、渲染、审计、或并行时，把它提升为专用工具。可逆性是个好判据——难以撤销的动作（外部 API 调用、发消息、删数据）应该是专用工具并挂审批。

miniclaude 的取舍：我们给一个 run_bash（广度）+ 三个专用文件工具 read_file / write_file / edit_file（因为文件操作要做路径安全校验和「写前必读」的新鲜度约束）。联网用 Anthropic 的服务端 web_search（第 8 节，无需本地执行）。

Anthropic 还内置了两个「无 schema」工具。bash_20250124 和 text_editor_20250728 是官方定义、由你客户端执行的工具——声明时只给 type 和 name，不要给 input_schema，模型内置了它们的用法。本文为了把「工具设计」讲透，用自定义工具演示；生产里直接用官方这两个也完全可以，且模型对它们的调用更熟练。

4定义工具与执行器

一个工具有两半：给模型看的定义（name/description/JSON Schema）和你来跑的执行器。先看定义。

tools.py — 工具定义python
TOOLS = [
    {
        "name": "read_file",
        "description": "读取工作区内一个文本文件的内容。在修改任何文件前，必须先用本工具读取它。",
        "input_schema": {
            "type": "object",
            "properties": {
                "path": {"type": "string", "description": "相对于工作区根目录的路径"}
            },
            "required": ["path"],
        },
    },
    {
        "name": "edit_file",
        "description": "对文件做精确字符串替换。old_str 必须在文件中唯一匹配一次，"
                       "否则报错。调用前必须先 read_file 读过该文件。",
        "input_schema": {
            "type": "object",
            "properties": {
                "path": {"type": "string"},
                "old_str": {"type": "string", "description": "要被替换的原文，需唯一"},
                "new_str": {"type": "string", "description": "替换后的新文本"},
            },
            "required": ["path", "old_str", "new_str"],
            "additionalProperties": False,
        },
    },
    {
        "name": "run_bash",
        "description": "在工作区内执行一条 bash 命令并返回 stdout+stderr。用于跑测试、"
                       "查目录、git 等。命令是不可信输入，会在隔离环境中受限执行。",
        "input_schema": {
            "type": "object",
            "properties": {"command": {"type": "string"}},
            "required": ["command"],
        },
    },
]
  1. description 是模型的唯一说明书。模型完全靠它决定何时调用。要规定式地写「什么时候用」，不只是「这是干嘛的」。注意 read_file 和 edit_file 的描述里都写了「修改前必须先读」——这是用自然语言给模型注入工作流约束，比事后报错更省一轮。近期 Opus 模型调工具更克制，把触发条件写进 description 能显著提升「该调就调」的命中率。
  2. input_schema 是标准 JSON Schema。用 required 标真正必填的参数；用 enum 锁定固定取值；每个属性都写 description。
  3. additionalProperties: False（见 edit_file）配合下面要讲的 strict 能保证模型不会塞进 schema 之外的字段。

执行器：把模型的意图变成真实副作用 —— 安全是第一位

path 和 command 都是不可信的模型输出。直接 open(path) 或 os.system(command) 是路径穿越和命令注入的活靶子。

tools.py — 安全的执行器python
import subprocess
from pathlib import Path

WORKSPACE = Path("./workspace").resolve()     # 固定的项目根，一切都关进这里

def _safe_path(rel: str) -> Path:
    """把模型给的相对路径解析为规范路径，并验证它仍在工作区内。"""
    target = (WORKSPACE / rel).resolve()
    if not target.is_relative_to(WORKSPACE):  # Python 3.9+: 拒绝 .. / 符号链接逃逸
        raise ValueError(f"路径越界，拒绝访问: {rel}")
    return target

def read_file(path: str) -> str:
    return _safe_path(path).read_text(encoding="utf-8")

def edit_file(path: str, old_str: str, new_str: str) -> str:
    p = _safe_path(path)
    text = p.read_text(encoding="utf-8")
    count = text.count(old_str)
    if count == 0:
        raise ValueError("old_str 未匹配到任何内容")
    if count > 1:
        raise ValueError(f"old_str 匹配到 {count} 处，必须唯一；请提供更多上下文")
    p.write_text(text.replace(old_str, new_str), encoding="utf-8")
    return f"已修改 {path}"

def run_bash(command: str) -> str:
    result = subprocess.run(
        ["bash", "-c", command],
        cwd=WORKSPACE, capture_output=True, text=True, timeout=120,
    )
    return f"[exit={result.returncode}]\n{result.stdout}\n{result.stderr}"

DISPATCH = {"read_file": read_file, "edit_file": edit_file, "run_bash": run_bash}
  1. WORKSPACE = ...resolve() 先把根目录规范化（解析符号链接、绝对化）。所有文件操作都必须落在这棵子树内。
  2. _safe_path 是安全核心。先把 WORKSPACE / rel .resolve() 成规范形式（这会展开 .. 和符号链接），再用 is_relative_to(WORKSPACE) 验证它没逃出根目录。../../etc/passwd、绝对路径、符号链接逃逸都会在这里被挡下。永远不要把原始 path 直接喂给 open()。
  3. edit_file 强制唯一匹配。0 处匹配或多处匹配都报错——这把「替换错地方」这个最常见的 Agent bug 在执行层挡掉，而不是寄希望于模型。报错信息写得有用（「匹配到 N 处，请提供更多上下文」），模型能据此自我纠正。
  4. run_bash 的命令是不可信输入。这里用 cwd=WORKSPACE 限定工作目录、timeout=120 限时。生产级还需要：跑在容器/受限用户里、可执行程序白名单、拒绝 shell 操作符（&& | ; `$()`）、资源限制、全量日志。黑名单不够。
  5. DISPATCH 把工具名映射到函数，循环里据此分发。

5核心：手写 agentic loop CORE

这是整个 Agent 的心脏。手写循环（而非用 Tool Runner）让你能在每一步插入日志、审批、条件执行——任何严肃产品都需要这种掌控力。逐行读。

agent.py — 手写 agentic looppython
import json
from client import client, MODEL
from tools import TOOLS, DISPATCH

SYSTEM = (
    "你是 miniclaude，一个在终端工作的自治编码助手。"
    "你能读写文件、运行命令。修改任何文件前必须先读取它。"
    "完成任务后用一两句话说明你做了什么。"
)

def run_agent(task: str) -> str:
    messages = [{"role": "user", "content": task}]

    while True:
        resp = client.messages.create(
            model=MODEL,
            max_tokens=16000,
            system=SYSTEM,
            thinking={"type": "adaptive"},
            tools=TOOLS,
            messages=messages,
        )

        if resp.stop_reason == "refusal":
            return "[模型出于安全原因拒绝了该请求]"

        if resp.stop_reason == "end_turn":
            return next(b.text for b in resp.content if b.type == "text")

        # 走到这里 stop_reason == "tool_use"：先把助手这一轮整段追加进历史
        messages.append({"role": "assistant", "content": resp.content})

        tool_results = []
        for block in resp.content:
            if block.type != "tool_use":
                continue
            try:
                output = DISPATCH[block.name](**block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": output,
                })
            except Exception as e:
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": f"工具执行失败: {e}",
                    "is_error": True,
                })

        messages.append({"role": "user", "content": tool_results})
  1. SYSTEM 提示词 定义 Agent 的人格与硬约束。对 Opus 4.8，别用「CRITICAL: YOU MUST」这种激进措辞——它会让模型过度触发；4.8 对系统提示词遵循度很高，平铺直叙即可。
  2. messages 是唯一的状态。API 无状态，整个对话历史靠这个数组在每次请求时全量带上。第一条必须是 user。
  3. thinking={"type": "adaptive"} 开启自适应思考：模型自己决定每一步想多深，并在工具调用之间自动穿插思考。这是 4.6+ 的推荐模式。注意 budget_tokens 在 Opus 4.8 上已移除，传了会 400——别用旧写法。要更省可加 output_config={"effort": "medium"}（low/medium/high/xhigh/max），但 Agent 类任务建议至少 high。
  4. 先判 refusal，再判 end_turn。安全分类器可能返回 HTTP 200 但 stop_reason == "refusal"，此时 content 可能为空。必须在读 resp.content[0] 之前先查 stop_reason，否则会在被拒请求上抛索引错误。
  5. end_turn = 模型说完了、不再要工具。从 content 里取出文本块返回，循环结束。next(...) 安全地拿第一个 type=="text" 的块——因为开了 thinking，content 里第一个块可能是 thinking 块，直接 content[0].text 会出错。
  6. 关键：先把 resp.content 整段追加进历史，再处理工具。这一步漏了，模型下一轮就看不到自己上一轮发的 tool_use 块，整个对话结构就坏了。必须追加整个 content（含 thinking 块、tool_use 块），不能只追加文本。
  7. 遍历所有 tool_use 块。一条助手消息里可能有多个 tool_use（并行工具调用，默认开启）。这里用 DISPATCH 分发执行。
  8. 每个 tool_result 必须带匹配的 tool_use_id。这是结果与调用的绑定键。漏一个 ID 没对上，下一轮请求会被 400 拒绝。
  9. 工具失败要回 is_error: True，而不是抛异常中断循环。把错误信息作为 tool_result 喂回去，模型会据此调整策略或换路。这让 Agent 有自愈能力。失败的工具不能丢——每个 tool_use 都必须有一个对应的 tool_result。
  10. 所有 tool_result 放进同一条 user 消息。千万别拆成多条 user 消息——那会悄悄训练模型以后不再做并行调用。一条 assistant（多个 tool_use）→ 一条 user（多个 tool_result）。

最容易写错的三处，按踩坑频率排序：① 忘了把 resp.content 追加进历史就去处理工具 → 模型「失忆」；② tool_result 漏 tool_use_id 或漏了某个块 → 400；③ 把多个 tool_result 拆进多条消息 → 并行能力退化。这三条几乎是每个新手都会犯的。

tool input 一律用结构化字段，别字符串匹配

注意上面用的是 block.input（SDK 已解析成 dict）和 **block.input 解包。Opus 4.8 可能对工具输入里的 JSON 字符串采用不同转义（Unicode、正斜杠），所以永远不要对序列化后的 input 做原始字符串匹配——SDK 给你的 .input 已经是解析好的对象，直接用。

6Tool Runner：少写一个循环 beta

如果你不需要在每一步插入审批/日志，SDK 的 Tool Runner 能把第 5 节那个循环整个吃掉。它自动：调 API → 检测 tool_use → 执行你的函数 → 喂回结果 → 重复，直到模型不再调工具。

用 @beta_tool 重写（对照）python
import anthropic
from anthropic import beta_tool

client = anthropic.Anthropic()

@beta_tool
def read_file(path: str) -> str:
    """读取工作区内一个文本文件。修改前必须先读。

    Args:
        path: 相对于工作区根目录的路径。
    """
    return _safe_path(path).read_text(encoding="utf-8")

# 同理用 @beta_tool 装饰 edit_file / run_bash ...

runner = client.beta.messages.tool_runner(
    model="claude-opus-4-8",
    max_tokens=16000,
    system=SYSTEM,
    tools=[read_file, edit_file, run_bash],
    messages=[{"role": "user", "content": task}],
)

for message in runner:        # 每圈 yield 一个 BetaMessage；模型不再调工具时自动停
    for block in message.content:
        if block.type == "text":
            print(block.text)
  1. @beta_tool 把一个带类型注解和 docstring 的普通函数变成工具：schema 从函数签名自动生成，description 来自 docstring。少维护一份手写 JSON Schema。
  2. client.beta.messages.tool_runner(...) 在 beta 命名空间下（这是 beta 功能）。返回一个可迭代的 runner。
  3. for message in runner: 每一圈产出一条完整的 BetaMessage，模型没有更多工具调用时迭代自动结束——你完全不用碰 tool_use_id、不用拼 tool_result、不用判 stop_reason。

怎么选：需要人审批准、自定义日志、条件执行、按工具并行调度 → 手写循环（第 5 节）。只是想跑通、工具无危险副作用 → Tool Runner。miniclaude 因为要做命令审批（第 13 节），主线用手写循环。

7流式 + Thinking/Effort CORE

交互式 Agent 必须流式：用户要看到「它在想、在动」，而不是盯着一个十几秒的空白。同时高 max_tokens 也强制要流式（第 2 节）。我们把第 5 节循环里的 create 换成 stream。

agent.py — 流式版的一圈python
with client.messages.stream(
    model=MODEL,
    max_tokens=16000,
    system=SYSTEM,
    thinking={"type": "adaptive", "display": "summarized"},
    output_config={"effort": "high"},
    tools=TOOLS,
    messages=messages,
) as stream:
    for event in stream:
        if event.type == "content_block_start":
            if event.content_block.type == "thinking":
                print("\n💭 思考中…", flush=True)
            elif event.content_block.type == "tool_use":
                print(f"\n🔧 调用 {event.content_block.name}", flush=True)
        elif event.type == "content_block_delta":
            if event.delta.type == "thinking_delta":
                print(event.delta.thinking, end="", flush=True)
            elif event.delta.type == "text_delta":
                print(event.delta.text, end="", flush=True)

    resp = stream.get_final_message()   # 流结束后拿到完整 Message，含 stop_reason
  1. with client.messages.stream(...) as stream: 推荐的流式入口（不是 create(stream=True)）。它会自动累积状态，并提供 get_final_message()。
  2. "display": "summarized" 必须显式加。在 Opus 4.8/4.7 上，thinking 的 display 默认是 "omitted"——思考块照样产生，但 thinking 字段是空串。如果你要把推理流给用户看，不加这一行就会表现为「一段长长的卡顿后才出文字」。
  3. output_config={"effort": "high"} effort 嵌在 output_config 里，不是顶层参数。它控制思考深度+整体 token 花销。Agent/编码任务官方建议 xhigh 最佳、至少 high；max 用在正确性压倒成本时。
  4. 逐事件消费。content_block_start 告诉你新块的类型（thinking / text / tool_use）；content_block_delta 带增量内容（thinking_delta / text_delta）。end="" + flush=True 让 token 即时显示。
  5. get_final_message() 是关键：流结束后它返回累积好的完整 Message，含 stop_reason 和所有 tool_use 块。第 5 节循环后续逻辑（判 stop_reason、追加历史、执行工具）原样接上。即使你只想要最终结果，也建议用 stream() + get_final_message()——它给你超时保护，又不用手动处理事件。

7.1 事件全景：一次流式响应长什么样

每个事件都有一个 SSE 事件名（event: ...）和一段 JSON data，data 里也带同名的 type。整条流的骨架是固定的：

事件流骨架flow
message_start                       # 1 次：一个 content 为空的 Message 壳
  ├─ content_block_start  (index=0)  # 块 0 开始：给出「块的骨架」
  │   ├─ content_block_delta ...     # 块 0 的增量（0 个或多个）
  │   └─ content_block_stop          # 块 0 结束
  ├─ content_block_start  (index=1)  # 块 1 开始……（按 index 顺序，一块完再下一块）
  │   ├─ content_block_delta ...
  │   └─ content_block_stop
  │   └─ ...
message_delta                       # 1 次或多次：顶层变更（stop_reason、累计 usage）
message_stop                        # 1 次：结束

# 另外可能随时穿插：ping（心跳，忽略）、error（如 overloaded_error）

每个内容块有一个 index，对应它在最终 Message.content 数组里的位置。标准模式下块是顺序产生的（一块 stop 后下一块才 start），index 用来标识它是谁。

7.2 关键心智模型：start 给「骨架」，delta 给「往某字段追加什么」

为什么 content_block_start 里有 tool_use，content_block_delta 里却没有 tool_use 相关的 delta？因为这两类事件回答的是不同的问题：

  • content_block_start 回答「这是一个什么块」→ 给你块的骨架：type，以及（对 tool_use）id / name，此时 input 还是空的 {}。
  • content_block_delta 回答「往这个块的哪个字段追加了什么」→ delta 的 type 表示正在被追加的字段，而不是块的类型。

一个 tool_use 块，唯一会增量变化的字段是 input，而 input 以 input_json_delta（partial_json 字符串片段）的形式流出。所以根本没有「tool_use_delta」——它就叫 input_json_delta。同理：text 块的增量是 text_delta，thinking 块的增量是 thinking_delta。delta 的类型 = 正在被填充的字段，不是块的类型。

把「块类型」和「它用哪种 delta 填充」对应起来，困惑就消失了：

块类型（在 content_block_start 里）用哪种 delta 填充追加到哪个字段
texttext_deltatext
tool_use / server_tool_useinput_json_delta（partial_json）input
thinkingthinking_delta，末尾一个 signature_deltathinking / signature
带 citations 的 texttext_delta + citations_deltatext / citations
redacted_thinking（无 delta，整块一次到达）data
web_search_tool_result 等服务端结果块（无 delta，整块一次到达）content
fallback（服务端 fallback 边界）（无 delta，start/stop 成对出现）—

7.3 每个事件携带什么字段

事件关键字段
message_startmessage：一个 content: [] 为空的 Message 壳，含 id / model / role:"assistant" / stop_reason:null / usage（此时 input_tokens 已知，output_tokens 约为 1，有缓存/服务端工具时还带 cache_*_input_tokens）

7.4 实战：一次 tool_use 的逐字 SSE（这是困惑的核心）

下面是模型决定调 get_weather 时的真实流（已加注释）。盯住 index=1 那个 tool_use 块：

tool_use 的真实 SSE 流（节选 + 注释）sse
event: content_block_start
data: {"type":"content_block_start","index":1,"content_block":{"type":"tool_use","id":"toolu_01T1...","name":"get_weather","input":{}}}
# ↑ 骨架在这里：type/id/name 都齐了，input 先是空 {}。「tool_use」只在 start 出现一次。

event: content_block_delta
data: {"type":"content_block_delta","index":1,"delta":{"type":"input_json_delta","partial_json":""}}
event: content_block_delta
data: {"type":"content_block_delta","index":1,"delta":{"type":"input_json_delta","partial_json":"{\"location\":"}}
event: content_block_delta
data: {"type":"content_block_delta","index":1,"delta":{"type":"input_json_delta","partial_json":" \"San"}}
event: content_block_delta
data: {"type":"content_block_delta","index":1,"delta":{"type":"input_json_delta","partial_json":" Francisc"}}
event: content_block_delta
data: {"type":"content_block_delta","index":1,"delta":{"type":"input_json_delta","partial_json":"o, CA\"}"}}
# ↑ input 不是一次给你一个对象，而是「JSON 字符串的碎片」。把所有 partial_json 顺序拼起来，
#   到 content_block_stop 时正好是合法 JSON：{"location": "San Francisco, CA"}

event: content_block_stop
data: {"type":"content_block_stop","index":1}
# ↑ 此刻才可以 json.loads(拼好的字符串) 得到最终 input 对象。

event: message_delta
data: {"type":"message_delta","delta":{"stop_reason":"tool_use","stop_sequence":null},"usage":{"output_tokens":89}}
# ↑ stop_reason 在这里揭晓；output_tokens 是整条响应的累计值。

event: message_stop
data: {"type":"message_stop"}
  • 为什么是「碎片字符串」而不是「一次给个对象」？为了最大粒度地流式。最终 tool_use.input 永远是个对象，但中途以 partial_json 字符串片段下发。当前模型一次发出一个完整的 key+value，所以工具参数流式时事件之间可能有停顿（模型在算下一个值）。
  • 怎么消费：按 index 把同一个块的所有 partial_json 顺序拼接，在该块的 content_block_stop 时再 json.loads。中途想要实时部分值，可用支持「部分 JSON 解析」的库（如 Pydantic）。
  • server_tool_use 同理。服务端工具（如 web_search）的调用也是 server_tool_use 块 + input_json_delta；紧接着它的结果会作为一个完整的 web_search_tool_result 块到达（只有 start/stop，无 delta，content 已填好）。

7.5 手工累积 vs SDK 帮你做

如果你直接消费原始事件，需要自己按 index 累积。但 99% 的情况下 SDK 已经替你做完了——这就是第 7 节开头用 stream() + get_final_message() 的原因：它把所有 partial_json 拼好并解析，最终 resp.content 里的 tool_use 块直接带解析好的 .input 对象。

两种写法对照python
# 写法 A：自己累积 partial_json（仅当你要逐字渲染工具参数时才需要）
buffers = {}                                   # index -> 拼接中的 JSON 字符串
with client.messages.stream(...) as stream:
    for event in stream:
        if event.type == "content_block_start" and event.content_block.type == "tool_use":
            buffers[event.index] = ""
            print(f"🔧 {event.content_block.name}", flush=True)   # 名字在 start 就有
        elif event.type == "content_block_delta" and event.delta.type == "input_json_delta":
            buffers[event.index] += event.delta.partial_json      # 累积碎片
    resp = stream.get_final_message()

# 写法 B（推荐）：让 SDK 累积，直接读解析好的 .input —— 第 5 节循环正是这么用的
with client.messages.stream(...) as stream:
    for event in stream:
        if event.type == "content_block_delta" and event.delta.type == "text_delta":
            print(event.delta.text, end="", flush=True)           # 只流式给用户看文本
    resp = stream.get_final_message()
for block in resp.content:
    if block.type == "tool_use":
        print(block.name, block.input)        # block.input 已是 dict，无需手动 json.loads

结论：要逐字展示工具参数 → 用写法 A 累积 partial_json；只想流式文本、拿到结果再处理工具 → 用写法 B，把累积交给 SDK。miniclaude 用 B。

7.6 thinking 块的流式与 signature

开了 thinking 且 display:"summarized" 时，思考块以 thinking_delta 流出，并在 content_block_stop 之前发一个特殊的 signature_delta——这是用于校验思考块完整性的签名。

thinking 块的 SSE（注释）sse
event: content_block_start
data: {"type":"content_block_start","index":0,"content_block":{"type":"thinking","thinking":"","signature":""}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"thinking_delta","thinking":"用欧几里得算法求 1071 和 462 的 GCD…"}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"signature_delta","signature":"EqQBCgIYAhIM1gbcDa9G..."}}
# ↑ stop 之前发一次签名。多轮对话里把 thinking 块回传时，signature 必须原样保留，不能改。

event: content_block_stop
data: {"type":"content_block_stop","index":0}
  • display:"omitted"（Opus 4.8 默认）时：不发任何 thinking_delta——块打开、收到一个 signature_delta、然后关闭。所以你看到的是「一段长停顿后才出正文」。要给用户看推理就显式设 display:"summarized"（第 7 节开头那行）。
  • signature 不可篡改。多轮把思考块回传给同一模型时，必须把 signature 原样带回；改了会被 API 拒绝。SDK 的 get_final_message() 会替你保留它。

7.7 message_delta：stop_reason 与累计 usage 的归宿

message_delta 是唯一告诉你顶层变更的地方：最终 stop_reason、stop_sequence，以及 usage。

usage.output_tokens 是累计值。每个 message_delta 给的是「到目前为止」的总输出 token，不是增量。要算最终用量直接取最后一个 message_delta 的值即可，不要把多个 message_delta 的 output_tokens 相加。服务端工具用量（如 server_tool_use.web_search_requests）也在末尾这条里。

用 get_final_message() 时这些都已归位：resp.stop_reason 和 resp.usage.output_tokens 直接读。这正是第 5 节循环里 if resp.stop_reason == ... 能成立的原因。

7.8 健壮性：ping / error / fallback / 未知事件

  • ping：心跳，散落在流中，直接跳过。
  • error：流中途也可能来错误（如高峰期的 overloaded_error，等价 HTTP 529）。要捕获并按可重试处理。用 SDK 的 with ... as stream 时，这些会以异常形式抛出，照第 13 节分层捕获即可。
  • fallback 块（仅服务端 fallback 场景）：在模型切换边界以一对 content_block_start/content_block_stop 出现，中间没有 delta。当成审计标记即可。
  • 未知事件类型：按官方版本策略，未来可能新增事件类型，你的代码应当优雅忽略不认识的事件，而不是崩溃。用 SDK 的高层流式助手能天然做到这点。

断流恢复（4.6+）：网络中断时，tool_use 和 thinking 块不能部分恢复——只能从最近一个完整的 text 块续。续跑方式在 4.6+ 上是发一条 user 消息「你上次的回复被中断，停在 […]，请接着写」，而不是用 assistant 前缀续（4.5 及更早才用 assistant 续）。优先依赖 SDK 的内置累积与重连。

8服务端工具与 pause_turn

有些工具跑在 Anthropic 的基础设施上，你不需要客户端执行：声明在 tools 里，结果作为内容块在同一次响应里返回。最常用的是 web_search（让 Agent 联网）和 code_execution（沙箱里跑代码）。

给 miniclaude 加联网能力python
TOOLS = [
    *TOOLS,                                                   # 第 4 节的自定义工具
    {"type": "web_search_20260209", "name": "web_search"},   # 服务端：联网搜索
]
  1. 服务端工具只声明、不执行。你在循环里不会看到需要你处理的 web_search 调用——Anthropic 替你执行，把 web_search_tool_result 块直接放进响应。miniclaude 的循环不用为它写任何分发逻辑。
  2. web_search_20260209 是带「动态过滤」的最新版（Opus 4.8/4.7/4.6、Sonnet 4.6 支持）：模型会写代码先过滤搜索结果再进上下文，更准更省 token。它内部已用到代码执行，所以不要再单独声明 code_execution，否则两个执行环境会让模型困惑。老模型用基础版 web_search_20250305。
  3. 服务端工具的错误不抛异常。它返回 HTTP 200，但结果块的 content 是个错误对象（如 {"error_code": "max_uses_exceeded"}）。成功时 content 是列表，错误时是对象——索引前先判断。

必须处理 pause_turn

服务端工具内部有自己的采样循环。如果它达到默认的 10 次迭代上限，响应会带 stop_reason == "pause_turn"。这不是结束，是「我暂停了，请让我接着跑」。在第 5 节循环里补一个分支：

在循环里处理 pause_turnpython
if resp.stop_reason == "pause_turn":
    messages.append({"role": "assistant", "content": resp.content})
    continue          # 原样再发一次，服务端会自动续跑——不要加「请继续」之类的 user 消息

不要在 pause_turn 后塞一条「Continue.」的 user 消息——API 检测到末尾的 server_tool_use 块就知道要续跑。再设一个 max_continuations（比如 5）防止无限循环。

9Prompt Caching 与前缀铁律

Agent 每一圈都把全量历史重新发一遍。一个跑了 30 轮的编码任务，前面的系统提示、工具定义、早期对话被重复处理几十次。Prompt Caching 让重复前缀以 ~0.1× 的价格命中，能省到 90%。

一条铁律解释一切：Prompt Caching 是前缀匹配。前缀里任何一个字节变化，都会让它后面所有内容的缓存失效。渲染顺序固定为 tools → system → messages。

给 system 加缓存断点python
resp = client.messages.create(
    model=MODEL,
    max_tokens=16000,
    system=[{
        "type": "text",
        "text": SYSTEM,                       # 庞大且稳定：人格 + 工具用法 + 代码规范
        "cache_control": {"type": "ephemeral"},
    }],
    tools=TOOLS,
    messages=messages,
)
print(resp.usage.cache_read_input_tokens)     # 命中缓存的 token（≈0.1× 价）
print(resp.usage.cache_creation_input_tokens) # 写入缓存的 token（≈1.25× 价）
  1. system 改成带 cache_control 的块列表。断点打在最后一个 system 块上，会把它前面渲染的 tools 和整个 system 一起缓存。多轮对话还可以把断点打在最近一轮 user 消息的最后一个内容块上，让历史前缀逐轮累积命中。
  2. 用 usage 验证命中。如果 cache_read_input_tokens 在相同前缀的重复请求里始终为 0，说明有「静默失效源」在作怪。

静默失效源——Agent 工程里最隐蔽的钱坑。这些放进前缀就让缓存全废：系统提示里塞 datetime.now() / UUID / 用户 ID；工具集随用户变化（tools 在位置 0，一变全废）；json.dumps 不带 sort_keys=True。系统提示要冻结，动态上下文（当前时间、模式）往 messages 后面塞，不要插进 system 头部。会话中途别加/删工具、别换模型——都会让整个缓存重建。

10长跑：Compaction 与 Context Editing

长跑 Agent 的历史会无限增长，最终撞上 1M 上下文窗口。两种官方机制，目的不同：

Compaction（压缩）Context Editing（编辑/清除）
做什么把早期历史总结成一个 compaction 块把陈旧的工具结果/思考块清除（删，不总结）
何时用对话可能接近/超过窗口上限老的工具输出已不相关，想保持轻量
beta headercompact-2026-01-12context-management-2025-06-27

Compaction

长会话开启服务端压缩python
resp = client.beta.messages.create(
    betas=["compact-2026-01-12"],
    model=MODEL,
    max_tokens=16000,
    messages=messages,
    context_management={"edits": [{"type": "compact_20260112"}]},

# 关键：必须把整段 response.content 追加回去，compaction 块要被保留
messages.append({"role": "assistant", "content": resp.content})

Compaction 的唯一致命坑：必须把 resp.content（整段，含 compaction 块）追加回 messages，而不是只追加文本。API 靠这个 compaction 块在下一次请求时替换掉被压缩的历史；只抽文本字符串追加会静默丢失压缩状态，下一轮直接把没压缩的历史又发上去。

Context Editing

清除陈旧的工具结果python
resp = client.beta.messages.create(
    betas=["context-management-2025-06-27"],
    model=MODEL, max_tokens=16000, tools=TOOLS, messages=messages,
    context_management={"edits": [{"type": "clear_tool_uses_20250919"}]},

策略类型：clear_tool_uses_20250919（清旧工具结果，加 clear_tool_inputs: true 连参数一起清）和 clear_thinking_20251015（清思考块）。别把它和 compaction 的类型搞混——compact_20260112 是另一个功能。

很多长跑 Agent 三个一起用：Context Editing 在会话内剪掉陈旧轮次，Compaction 在接近上限时总结，Memory（下一节）做跨会话持久化。

11跨会话记忆 memory

到目前为止 miniclaude 一关进程就失忆。Memory 工具让它把学到的东西写进一个文件目录，跨会话存活。它是客户端工具：模型发出 view/create/str_replace/insert/delete/rename 命令，你来实现存储后端。

声明 memory 工具python
TOOLS = [
    *TOOLS,
    {"type": "memory_20250818", "name": "memory"},   # 客户端工具，操作 /memories 目录
]

声明后，模型会用它在固定的 /memories 目录里读写文件。你在循环里收到 memory 的 tool_use 时，按命令读写文件并返回结果。Python SDK 提供了 BetaAbstractMemoryTool 辅助基类，子类化它实现六个方法即可，配合 Tool Runner 自动处理：

用 SDK 辅助基类实现存储后端python
from anthropic.lib.tools import BetaAbstractMemoryTool

class FileMemory(BetaAbstractMemoryTool):
    def view(self, command): ...

runner = client.beta.messages.tool_runner(

Memory 的安全要点，必踩：① 模型给的 path 是不可信输入，每次操作都要把它解析为规范路径并验证仍在 /memories 内，拒绝 ..、符号链接、绝对路径逃逸——和第 4 节 _safe_path 同样的校验。② 绝不在记忆文件里存 API key、密码、token。③ 多用户系统要做按用户隔离的记忆目录和鉴权——参考实现没有内置访问控制。

12结构化输出与子代理

结构化输出：让 Agent 的产物可被程序消费

当 Agent 要把结果交给下游代码（而不是给人读）时，用结构化输出约束 JSON 形状。推荐 messages.parse() + Pydantic，自动校验并返回类型化对象。

让 Agent 返回结构化的「任务报告」python
from pydantic import BaseModel

class TaskReport(BaseModel):
    summary: str
    files_changed: list[str]
    tests_passed: bool

resp = client.messages.parse(
    model=MODEL,
    max_tokens=16000,
    messages=[{"role": "user", "content": "总结你刚才做的修改"}],
    output_format=TaskReport,
)
report = resp.parsed_output          # 已校验的 TaskReport 实例
print(report.files_changed, report.tests_passed)

用 messages.create 时则写 output_config={"format": {"type": "json_schema", "schema": {...}}}。注意：旧的顶层 output_format 参数已废弃；结构化输出与 citations 互斥（同时用会 400）。

子代理：用便宜模型做子任务，又不毁缓存

主循环跑在 Opus 4.8 上。如果某个子任务（比如「在 200 个文件里搜一个符号」）不需要那么强的模型，不要会话中途换模型——那会让整个缓存失效。正确做法是派生一个子代理：用便宜模型（如 Haiku 4.5）单独起一次调用做子任务，主循环始终用一个模型。Claude Code 的 Explore 子代理就是这么用 Haiku 的。

同理，会话中途要扩展工具集时，别直接改 tools（位置 0，一改全废缓存），用 Tool Search 动态发现——它是追加工具 schema 而非替换，保住缓存前缀。

13错误处理、refusal 与人审

分层捕获异常，而不是一把抓

一个 except APIError 会丢掉「可重试（429/5xx/网络）」和「不可重试（400/404）」的区别。按从具体到一般的顺序链式捕获：

健壮的请求封装python
import anthropic

try:
    resp = client.messages.create(...)
except anthropic.NotFoundError:           # 404：多半是 model ID 写错
    raise
except anthropic.RateLimitError as e:     # 429：SDK 已自动退避重试；到这说明重试也耗尽
    retry_after = int(e.response.headers.get("retry-after", "60"))
    ...
except anthropic.APIStatusError as e:     # 其他非 2xx
    if e.status_code >= 500:
        ...                               # 服务端错误，可重试
    else:
        raise                             # 4xx 客户端错误，不该重试
except anthropic.APIConnectionError:      # 网络层失败
    ...

SDK 已经自动重试 429 和 5xx（指数退避，默认 max_retries=2，我们在第 2 节调到了 4）。只有当你需要超出 SDK 默认行为时才手写重试逻辑。

refusal：在读 content 之前就要处理

第 5 节已经把 refusal 分支放在循环最前面。重申原因：安全分类器可能返回 HTTP 200 + stop_reason == "refusal"，且 content 可能为空。stop_details 仅在 refusal 时非空，其他 stop_reason 下是 None——读 .category 前要先判 stop_reason。

人审：把危险动作挡在执行前

这是手写循环（而非 Tool Runner）的核心价值。run_bash 可能跑 rm -rf 或 git push。在执行前插入审批门：

agent.py — 在分发执行前加审批python
DANGEROUS = {"run_bash", "edit_file"}

for block in resp.content:
    if block.type != "tool_use":
        continue
    if block.name in DANGEROUS:
        print(f"\n⚠️  Agent 想执行 {block.name}: {block.input}")
        if input("批准？[y/N] ").strip().lower() != "y":
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": "用户拒绝了该操作。请换一种方式或询问用户。",
                "is_error": True,
            })
            continue
    output = DISPATCH[block.name](**block.input)
    tool_results.append({"type": "tool_result", "tool_use_id": block.id, "content": output})
  1. 审批发生在 DISPATCH 之前——这就是「把动作提升为专用工具」（第 3 节）的回报：因为 run_bash/edit_file 是带类型参数的专用工具，你能在执行前看清它要做什么并拦下。如果一切都走一个不透明的 bash 字符串，你根本没这个把手。
  2. 拒绝时回 is_error + 有用的说明。把「用户拒绝了，请换方式」喂回去，模型会调整策略，而不是傻等或崩溃。

14完整整合：能跑的 CLI agent CORE

把前面所有东西拼成一个能跑的 main.py：流式 + thinking/effort + 缓存 + 工具循环 + 审批 + pause_turn + refusal + 多轮会话。这就是 miniclaude 的全部。

main.py — 完整的 miniclaudepython
import anthropic
from tools import TOOLS, DISPATCH       # 第 4 节

MODEL = "claude-opus-4-8"
client = anthropic.Anthropic(max_retries=4, timeout=600.0)

SYSTEM = [{
    "type": "text",
    "text": (
        "你是 miniclaude，一个在终端工作的自治编码助手。"
        "你能读写文件、运行命令、必要时联网搜索。"
        "修改任何文件前必须先 read_file。完成后用一两句话汇报你做了什么。"
    ),
    "cache_control": {"type": "ephemeral"},     # 冻结的系统提示 → 可缓存
}]

DANGEROUS = {"run_bash", "edit_file"}

def step(messages):
    """跑一圈：流式请求 + 落地工具，返回 (final_text 或 None, 是否结束)。"""
    with client.messages.stream(
        model=MODEL, max_tokens=16000, system=SYSTEM,
        thinking={"type": "adaptive", "display": "summarized"},
        output_config={"effort": "high"},
        tools=TOOLS, messages=messages,
    ) as stream:
        for event in stream:
            if event.type == "content_block_delta" and event.delta.type == "text_delta":
                print(event.delta.text, end="", flush=True)
        resp = stream.get_final_message()

    if resp.stop_reason == "refusal":
        return "[模型出于安全原因拒绝了请求]", True
    if resp.stop_reason == "end_turn":
        return next((b.text for b in resp.content if b.type == "text"), ""), True
    if resp.stop_reason == "pause_turn":                 # 服务端工具续跑
        messages.append({"role": "assistant", "content": resp.content})
        return None, False

    # stop_reason == "tool_use"
    messages.append({"role": "assistant", "content": resp.content})
    results = []
    for block in resp.content:
        if block.type != "tool_use":
            continue
        if block.name in DANGEROUS:
            print(f"\n⚠️  执行 {block.name}: {block.input}")
            if input("批准？[y/N] ").strip().lower() != "y":
                results.append({"type": "tool_result", "tool_use_id": block.id,
                                "content": "用户拒绝。请换方式。", "is_error": True})
                continue
        try:
            out = DISPATCH[block.name](**block.input)
            results.append({"type": "tool_result", "tool_use_id": block.id, "content": out})
        except Exception as e:
            results.append({"type": "tool_result", "tool_use_id": block.id,
                            "content": f"失败: {e}", "is_error": True})
    messages.append({"role": "user", "content": results})
    return None, False

def main():
    messages = []
    print("miniclaude 已就绪（Ctrl-C 退出）")
    while True:
        task = input("\n\n你 ▸ ")
        messages.append({"role": "user", "content": task})
        done = False
        while not done:                                  # agentic loop
            final, done = step(messages)
        print(f"\n\n✓ {final}")

if __name__ == "__main__":
    main()
  1. step() 是「一圈」。它发一次流式请求、流式打印文本、用 get_final_message() 拿完整响应，然后按 stop_reason 四向分支：refusal/end_turn 结束，pause_turn 续跑，tool_use 执行工具。返回 (最终文本或None, 是否结束)。
  2. 四个 stop_reason 全覆盖。这正是新手最常漏的——只处理 tool_use 和 end_turn，结果在服务端工具（pause_turn）或安全拒绝（refusal）上崩掉。
  3. 外层 main() 维护多轮会话。messages 跨用户输入持续累积——用户第二句话时，模型还记得第一句和中间所有工具结果。每个 step 都把助手响应和工具结果追加进同一个 messages，所以历史是连续的。
  4. 系统提示带 cache_control。因为它冻结不变，几十圈循环里它和工具定义只在第一圈付全价，后续都命中缓存。

什么时候该换成 Managed Agents（路径 B）？当你需要的不是「装进自己进程」，而是「让 Anthropic 替我跑循环 + 提供每会话一个沙箱容器」时：长跑研究 Agent、要把 SSE 事件流推给前端 UI、要持久化/版本化的 Agent 配置、要按会话挂载文件/仓库。那时改用 client.beta.agents.create() 建一个持久 Agent，再 sessions.create() 起会话。但只要你想掌控每次工具执行和审批，本文这套自托管循环就是对的选择。

15最佳实践清单 & 速查

循环正确性（最高频的 bug）

  • 每一圈先把整段 resp.content 追加进 messages，再处理工具。
  • 每个 tool_result 必带匹配的 tool_use_id；失败的工具也要回结果（is_error: True），不能丢。
  • 一条 assistant（多 tool_use）→ 一条 user（多 tool_result）。别拆成多条。
  • 四个 stop_reason 全处理：end_turn / tool_use / pause_turn / refusal（外加 max_tokens）。
  • 读 resp.content 前先判 refusal；用 next(b.text for b ... if b.type=="text") 而非 content[0].text（开了 thinking 时首块可能是 thinking）。

模型参数

  • 模型 ID 用完整即用字符串 claude-opus-4-8，不补日期后缀。
  • Thinking 用 {"type": "adaptive"}；要给用户看推理加 "display": "summarized"。不要用 budget_tokens（4.8 上传了 400）。
  • Effort 在 output_config 里，Agent/编码任务至少 high，xhigh 最佳。
  • 输出可能 > ~16K token → 用 messages.stream()；非流式默认 max_tokens=16000，流式可给 64000。

安全

  • path / command 是不可信输入：路径做规范化+根目录校验；命令白名单+隔离+限时+日志。
  • 难以撤销的动作提升为专用工具并挂人审。
  • 记忆/工具不存任何密钥；多用户做按用户隔离 + 鉴权。

上下文与成本

  • 系统提示冻结、工具集稳定、JSON 确定性序列化——保住前缀缓存。
  • 长跑用 Context Editing（剪）+ Compaction（总结），后者必须把整段 content（含 compaction 块）追加回去。
  • 子任务用便宜模型时派生子代理，别在主会话里换模型/换工具集。

工程

  • 用官方 SDK，别手搓 HTTP；用 SDK 的类型、流式助手、分层异常类。
  • 不需要每步审批/日志时，用 Tool Runner 省掉整个循环。
  • 需要托管循环 + 沙箱容器时，转 Managed Agents。
