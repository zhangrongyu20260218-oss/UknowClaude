Claude Managed Agents 完整讲解

把 Agent 配置、沙箱、会话、事件流、权限、多代理与调度交给托管运行时 · 面向资深程序员

时效声明：本文按 2026-06-21 的官方文档整理。Managed Agents 仍是 beta，所有端点要求 managed-agents-2026-04-01 beta header；SDK 会自动设置。字段和限制可能继续变化。

一句话心智模型：Messages API 给你模型调用，Claude Agent SDK 给你可嵌入的 agent harness，Managed Agents 再把 harness、沙箱和长跑生命周期变成服务端资源。你不再手写 while-loop、容器调度、文件检查点和断线恢复，而是围绕 Agent → Environment → Session → Events 四个对象编程。

1它到底托管了什么

Managed Agents 不是另一个聊天接口，也不是“把 prompt 存到云端”。它是一个有状态、可暂停、可恢复、带安全沙箱的 Agent 运行面：

Agent模型、system、工具、MCP、Skills
Environment运行位置、包、网络与沙箱
Session一次长跑任务及持久状态
Events输入、输出、工具、人审与状态
你的应用 │ create / send event / stream / webhook ▼ Managed Agents control plane ├─ Agent definition (versioned) ├─ Environment definition └─ Session state machine ├─ conversation history ├─ isolated Linux sandbox + checkpoint ├─ agent loop + prompt cache + compaction └─ tools / MCP / custom tool rendezvous

Claude 会在沙箱内自主读写文件、执行 Bash、搜索网页、调用 MCP。你的应用负责定义边界、发任务、处理需要宿主执行的 custom tool、审批高风险动作、接收事件和交付产物。

托管的是执行基础设施，不是业务责任。工具权限、网络 allowlist、租户凭证、幂等、预算、rubric 质量和最终副作用仍然属于你的系统。

2三种构建层级怎么选

方案你掌控什么适合代价
Messages API + 手写 loop每次模型调用、上下文、工具调度、存储强合规、特殊调度、极致可控循环、重试、沙箱、恢复都自己造
Claude Agent SDK在你的进程/基础设施内使用现成 harness本地工具、嵌入已有服务、自己运维仍需承接运行环境与长跑生命周期
Managed AgentsAgent 配置与策略；Anthropic 承接运行时分钟到小时、异步、产物型工作流服务端状态、beta 变化、额外 runtime 计费

优先选 Managed Agents 的信号

  • 任务需要多轮工具调用，运行几分钟到几小时，而 HTTP 请求不该一直挂住。
  • 需要隔离 Linux 容器、文件系统、预装包、网络限制和 checkpoint。
  • 用户会中途补充要求、批准工具、暂停后恢复，或者最终下载一组文件。
  • 希望用 webhook、cron、多代理和 outcome grader 做工作流，而不是聊天框。

不该选的信号

  • 一次请求就能完成的分类、抽取、短问答；Messages API 更直接、更便宜。
  • 要求 Zero Data Retention 或 HIPAA BAA 覆盖。当前 Managed Agents 不符合这两项。
  • 必须在合作云的 Bedrock / Vertex AI 上跑；当前产品面不是这两个平台的功能。
  • 必须精确控制每一轮 prompt 和每个并行工具的调度顺序。

3最小可运行主线 CORE

安装最新 Anthropic SDK，设置 ANTHROPIC_API_KEY。SDK 自动带 beta header；直接 curl 才需要手动带。

shellpython -m pip install -U anthropic
export ANTHROPIC_API_KEY="sk-ant-..."

最小流程不是“建 Agent 然后 run”，而是创建三个资源，再打开事件流并发送用户事件：

pythonimport anthropic

client = anthropic.Anthropic()

agent = client.beta.agents.create(
    name="Coding Assistant",
    model="claude-opus-4-8",
    system="You are a careful coding agent. Verify every change.",
    tools=[{"type": "agent_toolset_20260401"}],
)

environment = client.beta.environments.create(
    name="coding-python",
    config={
        "type": "cloud",
        "networking": {"type": "limited", "allowed_hosts": []},
        "packages": {"pip": ["pytest"]},
    },
)

session = client.beta.sessions.create(
    agent=agent.id,
    environment_id=environment.id,
    title="Build and test fibonacci.py",
)

with client.beta.sessions.events.stream(session.id) as stream:
    client.beta.sessions.events.send(
        session.id,
        events=[{
            "type": "user.message",
            "content": [{
                "type": "text",
                "text": "Create fibonacci.py, add tests, run pytest, and summarize the result.",
            }],
        }],
    )

    for event in stream:
        match event.type:
            case "agent.message":
                for block in event.content:
                    if hasattr(block, "text"):
                        print(block.text, end="")
            case "agent.tool_use":
                print(f"\n[tool: {event.name}]")
            case "session.status_idle":
                reason = event.stop_reason
                if reason and reason.type == "end_turn":
                    break

最常见的第一个 bug：看到 session.status_idle 就 break。idle 也可能是 requires_action，表示它在等 custom tool 结果或人工确认；必须检查 stop_reason.type。

4Agent：版本化配置，不是运行实例

Agent 把模型、system、tools、MCP servers、skills、multiagent roster 与 metadata 封成可复用资源。Session 才是运行实例。

字段含义工程注意
modelClaude 4.5 系列及以后生产上显式记录模型与 Agent version
system固定行为与角色任务本身放 user event，不要塞这里
tools内置 / MCP / custom tools更新时数组全量替换
skills按需加载的文件型知识每个 session 全体 agent 合计最多 20 个
multiagent协调者可委派的 roster引用版本在协调者更新时快照固定

更新语义是乐观并发 + 版本化

pythonupdated = client.beta.agents.update(
    agent.id,
    version=agent.version,  # 防止覆盖别人刚做的更新
    system="You are a careful coding agent. Always run tests.",
)
  • 标量字段替换；tools / mcp_servers / skills 是整数组替换。
  • 无实际变化不会创建新版本。
  • 用字符串 Agent ID 创建 Session 会取当时 latest；要灰度就传 {id, version} 固定版本。
  • 归档后新 Session 不能再引用，但已有 Session 可继续。

不要把 Agent update 当配置热更新。已有 Session 不会自动变成新 Agent 版本；Session 只允许在 idle 时局部替换自己的 tools / MCP 配置，而且仍是数组全量替换。

5Environment：可复用配置，每个 Session 独立沙箱

多个 Session 可以引用同一个 Environment，但每个 Session 都得到新的隔离 Linux 容器，不会共享文件系统。共享的是构建配置和包缓存。

pythonenvironment = client.beta.environments.create(
    name="report-prod-v3",
    config={
        "type": "cloud",
        "packages": {
            "apt": ["ffmpeg"],
            "pip": ["pandas==2.2.0", "openpyxl"],
        },
        "networking": {
            "type": "limited",
            "allowed_hosts": ["api.example.com"],
            "allow_package_managers": False,
            "allow_mcp_servers": True,
        },
    },
)

默认运行面

  • Ubuntu 22.04 x86_64，最高 8 GB 内存、10 GB 磁盘。
  • Python 3.12+、Node 20+、Go 1.22+、Rust 1.77+、Java 21+ 等预装。
  • Environment 不版本化。推荐名字带版本，并在业务库记录 session → environment 配置摘要。
  • 官方页面存在一处容易误读：参考页写“network disabled by default”，环境配置页写 unrestricted 是 networking 默认值。生产代码不要依赖默认值，务必显式传 limited。

生产基线：limited + 最小 host allowlist；包在 Environment 构建期固定，不要让 Agent 运行中随意 pip install latest。

6Session：必须按状态机写 CORE

状态含义你的动作
idle等待消息、审批或 custom tool 结果检查 stop_reason，别盲目判完成
runningAgent 正在执行流式消费；可 steer / interrupt
rescheduling暂时错误，平台自动重试观察，不要立即重复创建任务
terminated不可恢复错误读取 session.error，决定新建 Session
create session ──► idle │ user.message / define_outcome ▼ running ◄────────────┐ │ │ tool result / confirmation ┌───────────┼───────────┐ │ ▼ ▼ ▼ │ idle:end_turn idle:requires_action rescheduling │ │ │ 完成/续聊 宿主处理事件 ─────────┘ │ unrecoverable ▼ terminated

持久化边界

会话历史保留到显式删除；idle 时沙箱被 checkpoint，文件、安装包和产物可恢复。但 checkpoint 只保证最后活动后的 30 天。跨月业务不能把 Session checkpoint 当永久对象存储。

  • 归档：禁止再发事件，保留历史。
  • 删除：永久删除 Session、events 和关联 sandbox。
  • Files、Vaults、Skills、Agents、Environments 是独立资源，不会随 Session 删除。

7Events 与流式：把 SSE 当可重放日志

事件名采用 {domain}.{action}。你发送 user.* / system.*，平台返回 agent.* / session.* / span.*。每个事件都有 processed_at；为 null 表示已排队、尚未处理。

生产消费器的四条规则

  1. 按 event ID 去重并落库，断线后 list 历史事件补洞。
  2. session.status_idle 必须分支处理 end_turn 与 requires_action。
  3. 收到 session.error 记录原始事件和 Session ID，不能只展示“失败”。
  4. 把 UI 文本流和业务状态分开；文本可以丢一小段，审批/工具事件不能丢。
pythondef consume(session_id: str):
    seen = set()
    events_by_id = {}

    with client.beta.sessions.events.stream(session_id) as stream:
        for event in stream:
            if event.id in seen:
                continue
            seen.add(event.id)
            events_by_id[event.id] = event
            persist(event)

            if event.type == "session.error":
                raise_session_alert(session_id, event)

            if event.type != "session.status_idle":
                continue

            reason = event.stop_reason
            if reason.type == "end_turn":
                return "done"
            if reason.type == "requires_action":
                for blocked_id in reason.event_ids:
                    dispatch_required_action(events_by_id[blocked_id])

中途 steer 与 system.message

Session running 时可以追加用户事件改变方向。system.message 用于轮次之间更新系统级约束，但当前只支持 Opus 4.8；且 Session 正因 requires_action idle 时不能发送。

这和 Messages API 的“重发完整 messages 数组”完全不同：Managed Agents 在服务端保存 event history，你操作的是追加式日志。

8工具、权限与宿主回调

内置工具

agent_toolset_20260401 默认启用 Bash、Read、Write、Edit、Glob、Grep、Web Fetch、Web Search。工具输出超过 100,000 tokens 时会写入沙箱文件，模型只收到预览和文件路径。

json{
  "type": "agent_toolset_20260401",
  "default_config": {
    "enabled": false,
    "permission_policy": {"type": "always_ask"}
  },
  "configs": [
    {"name": "read", "enabled": true, "permission_policy": {"type": "always_allow"}},
    {"name": "grep", "enabled": true, "permission_policy": {"type": "always_allow"}},
    {"name": "write", "enabled": true},
    {"name": "bash", "enabled": true}
  ]
}
工具来源在哪里执行默认审批回填事件
内置 agent toolsetManaged sandboxalways_allow需要时 user.tool_confirmation
MCP toolset远端 MCP serveralways_askuser.tool_confirmation
Custom tool你的应用不受 policy 管user.custom_tool_result

最危险的默认值是内置 toolset 的 always_allow。一旦同时给了 unrestricted networking 和 Bash，模型就有广泛外部副作用能力。生产上应默认 deny/ask，再逐项开放只读工具。

Custom tool 是一次 rendezvous

  1. 平台发 agent.custom_tool_use。
  2. Session 进入 idle / requires_action。
  3. 你的 worker 校验 schema、租户、幂等键、权限并执行。
  4. 发送 user.custom_tool_result，所有阻塞项解决后恢复 running。

9文件、GitHub、MCP、Skills 与 Vault

文件：上传一次，Session 里挂载副本

pythonfrom pathlib import Path

source = client.beta.files.upload(file=Path("requirements.pdf"))
session = client.beta.sessions.create(
    agent=agent.id,
    environment_id=environment.id,
    resources=[{
        "type": "file",
        "file_id": source.id,
        "mount_path": "/workspace/requirements.pdf",
    }],
)

单 Session 最多挂 100 个文件。输出约定放 /mnt/session/outputs/，Session idle 后按 scope_id=session.id 列出和下载。

GitHub：repo mount 和 GitHub MCP 是两条通道

  • github_repository resource 负责 clone/mount 代码。
  • GitHub MCP 负责 issue / PR 等 API 动作。
  • clone token 放 Session resource；MCP OAuth 更适合放 Vault。两者不要混为一谈。
  • 仓库有缓存，同一 repo 的后续 Session 启动更快。

Vault：每个最终用户一组凭证

Vault 是 workspace-scoped 资源，Session 创建时传 vault_ids。支持 MCP OAuth、static bearer 等凭证，平台可代管 OAuth refresh。任何同 workspace API key 都能引用 Vault，所以要在你的业务层做 user → vault 映射和授权。

Skills：按需加载的文件型专业知识

Skills 和大 system prompt 的区别是 progressive disclosure：只有任务相关时才进入上下文。可挂 Anthropic 预置的 xlsx / pptx / docx / pdf 技能，或自建版本化 Skill。

10Outcomes：从“聊完”升级为“验收通过” 研究预览

user.define_outcome 同时给目标、Markdown rubric 和最大迭代数。Harness 会创建一个独立上下文的 grader，评价产物并把反馈交回 Agent 继续修改。

pythonclient.beta.sessions.events.send(
    session.id,
    events=[{
        "type": "user.define_outcome",
        "description": "Build a tested Python package and write its README.",
        "rubric": {
            "type": "text",
            "content": """# Correctness
- pytest passes
- public functions have type hints
# Deliverables
- package and README exist under /mnt/session/outputs/""",
        },
        "max_iterations": 5,  # 默认 3，最大 20
    }],
)

关键事件是 span.outcome_evaluation_start/ongoing/end。结束结果有 satisfied、needs_revision、max_iterations_reached、failed、interrupted。

Outcome 不是形式化验证。Grader 仍是模型，rubric 含糊会得到含糊评价。能用 pytest、schema validator、编译器、财务勾稽等确定性工具验证的条件，应该先由工具产出硬证据，再让 grader 做语义判断。

11多代理：共享沙箱，隔离上下文

Coordinator 的 multiagent.agents 声明可委派 roster。子代理有各自 Agent 配置和事件线程，但共享同一 Session 的 sandbox、filesystem 与 Vault credentials。

pythoncoordinator = client.beta.agents.create(
    name="Engineering Lead",
    model="claude-opus-4-8",
    system="Delegate security review and test design, then synthesize.",
    tools=[{"type": "agent_toolset_20260401"}],
    multiagent={
        "type": "coordinator",
        "agents": [
            {"type": "agent", "id": reviewer.id, "version": reviewer.version},
            {"type": "agent", "id": test_writer.id},
        ],
    },
)
  • 最多列 20 个 unique agents；只能委派一层，depth > 1 被忽略。
  • 最多 25 个并发 threads；同一 Agent 可启动多个副本。
  • primary event stream 只给压缩视图；排障要进入具体 thread stream。
  • roster 在 Coordinator 创建/更新时固定版本，不会自动跟随子 Agent latest。
  • MCP servers 是 Agent-scoped；Vault credentials 是 Session-scoped，所有线程可引用。

适合并行调研、专业分工、能力升级。不适合把一个线性任务硬拆成多 Agent；协调成本和 token 会真实增加。

12Webhook 与定时任务：把长跑接进产品

SSE 用于实时，Webhook 用于状态通知

Webhook 只发 event type 和资源 ID，不带完整对象；收到后再 GET，避免 retry 投递陈旧快照。签名头是 X-Webhook-Signature，SDK unwrap() 同时验签和拒绝超过 5 分钟的 payload。

  • 用顶层 event ID 幂等；同 ID 是重试。
  • 不保证顺序，按 created_at 排，不按到达时间猜。
  • 任何 3xx 都算失败；约 20 次连续失败会自动 disable endpoint。
  • Webhook 必须是公网 HTTPS 443，不能解析到私网 IP。

Scheduled deployments

Deployment 把 Agent、Environment、resources、Vault 和首条 user.message 固定成 cron。使用五段 POSIX cron + IANA timezone，分钟级粒度，触发可能有最多 10 秒 jitter。每组织最多 1,000 个 deployment。

调度的是“每次新建一个 Session”，不是唤醒同一 Session。需要累积记忆时，应显式挂 Memory Store 或把上次结果作为文件/数据库资源输入。

13成本、限制与数据边界

两维计费

项目计费
模型 tokens按所选 Claude 模型价格；prompt cache 倍率相同
Session runtime$0.08 / session-hour，只计 running，毫秒级计量
Web search$10 / 1,000 searches + 搜索内容 token
Idle / rescheduling / terminated不计 runtime

Managed Agents 没有 Batch 折扣，也不套 fast mode premium、inference_geo multiplier 或长上下文 premium。Session runtime 已替代 code-execution container-hour，不再重复收容器费。

固定限制速查

  • Create endpoints：300 RPM / organization；Read/list/stream：600 RPM。
  • 每 Session 最多 100 个挂载文件、20 个 Skills、25 个并发 Agent threads。
  • 每 Coordinator roster 最多 20 个 unique agents，且只支持一层委派。
  • 每组织最多 1,000 个 scheduled deployments。
  • checkpoint 最后活动后保留 30 天；event history 到删除为止。

数据与合规

由于服务端持久化会话历史、沙箱状态和输出，Managed Agents 当前不支持 Zero Data Retention，也不在 HIPAA BAA 覆盖内。删除 Session 不会级联删除独立上传的 Files、Vaults、Skills、Agents 或 Environments。

14生产实现骨架：控制面与事件 worker 分离

POST /jobs ├─ validate tenant / budget / agent version ├─ create Session (idempotency key in your DB) ├─ persist job_id ↔ session_id └─ send user.message or user.define_outcome event worker ├─ stream SSE; reconnect + list history on gaps ├─ persist event IDs ├─ dispatch custom tools / approvals ├─ update user-facing job state └─ on end_turn: list outputs + notify webhook worker ├─ unwrap signature ├─ dedupe webhook event ID └─ GET latest Session before acting

业务表至少记录这些

字段为什么
tenant_id / user_id / vault_id防止跨租户引用凭证
agent_id / agent_version / environment_id可复现与灰度回滚
session_id / last_event_id恢复消费和排障
status / stop_reason / error区分完成、待审批、平台重试、终止
usage / runtime / budget成本保护
output_file_ids交付与生命周期清理

可靠性底线

  • 创建 Job 先落库再调 API；网络超时后先按业务幂等键查，别盲建第二个 Session。
  • Custom tool 自己做 schema 校验、超时、幂等和审计；工具描述不是安全边界。
  • 对 running 设置业务级 deadline，超时发 interrupt，不让任务静默烧 token。
  • 产物复制到你自己的对象存储；30 天 checkpoint 不是归档策略。
  • 记录原始 event，不只记录拼接后的聊天文本。
